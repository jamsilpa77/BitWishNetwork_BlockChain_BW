현재 대부분의 블록체인들은 **3개의 계층 구조**로 데이터를 저장.

---

# 1. 블록체인의 기본 저장 구조 (3계층)

블록체인은 보통 다음 구조로 저장된다.

1️⃣ **블록 데이터 (Block Data)**
2️⃣ **트랜잭션 데이터 (Transaction Data)**
3️⃣ **상태 데이터 (State Database / State Trie)**

이 3개가 함께 동작한다.

---

# 2. 블록 데이터 저장 방식

블록체인의 가장 기본은 **블록 체인 구조**이다.

각 블록은 다음 데이터를 포함한다.

```
Block
 ├ Block Header
 │  ├ previous_block_hash
 │  ├ state_root
 │  ├ transaction_root
 │  ├ timestamp
 │  └ block_number
 │
 └ Transactions
    ├ tx1
    ├ tx2
    ├ tx3
```

여기서 중요한 것이 있다.

**state_root**
**transaction_root**

이 두 개가 **트리 구조의 해시 루트**이다.

---

# 3. 트랜잭션 저장 방식 (Merkle Tree)

트랜잭션은 보통 **Merkle Tree**로 저장된다.

예시

```
        Root
       /    \
    H12      H34
   /  \     /   \
 H1   H2  H3    H4
```

각 트랜잭션을 해시한다.

```
tx1 -> H1
tx2 -> H2
tx3 -> H3
tx4 -> H4
```

그 후 위로 합쳐서 루트를 만든다.

이 방식의 장점

* 데이터 위변조 방지
* 부분 검증 가능 (SPV)
* 빠른 검증

이 구조는 거의 **모든 블록체인이 사용한다.**

대표 예

* Bitcoin
* Ethereum

---

# 4. 상태 저장 방식 (State Trie)

현대 블록체인의 핵심은 **상태(State)** 이다.

예를 들어

```
A = 100 코인
B = 50 코인
C = 10 코인
```

이것이 현재 상태이다.

하지만 블록체인은 **상태를 직접 테이블로 저장하지 않는다.**

대신

**Merkle Patricia Trie**

라는 구조를 사용한다.

---

# 5. Merkle Patricia Trie 구조

이것은 **Trie + Merkle Tree** 구조이다.

예시

```
          Root
          / \
        node node
        /      \
     account   account
```

각 계정 정보는 다음과 같이 저장된다.

```
account
 ├ address
 ├ balance
 ├ nonce
 ├ storageRoot
 └ codeHash
```

그리고 이 모든 노드들은 **해시로 연결된다.**

그래서 최종적으로

```
State Root
```

가 생성된다.

이 값이 블록 헤더에 저장된다.

---

# 6. 실제 블록체인 노드의 저장 구조

실제 노드는 다음 구조로 데이터를 저장한다.

```
NODE STORAGE
│
├ Block DB
│   ├ block_header
│   └ block_body
│
├ Transaction DB
│
├ State DB
│   └ Merkle Patricia Trie
│
└ Index DB
```

여기서 중요한 점이 있다.

**State DB는 트리 구조지만 실제 저장은 Key-Value DB이다.**

대표적으로 사용되는 DB

* LevelDB
* RocksDB

예

```
key: hash
value: node_data
```

즉

**트리는 논리 구조**
**DB는 실제 저장 구조**

입니다.

---

# 7. MongoDB 방식과 블록체인 방식 차이

MongoDB 방식

```
accounts table
transactions table
blocks table
```

이 방식은

* 중앙 서버
* 수정 가능
* 관리자 존재

이다.

반면 블록체인은

```
hash tree
state root
block chain
```

특징

* 수정 불가능
* 모든 노드 동일 데이터
* 검증 가능 구조

---

# 8. 블록체인이 MongoDB 전체 저장을 안하는 이유

만약 MongoDB 방식으로 저장하면 문제가 생긴다.

문제

1️⃣ 데이터 위변조 검증 불가능
2️⃣ 노드간 데이터 일치 검증 불가능
3️⃣ 부분 검증 불가능

그래서 반드시

**해시 트리 구조**

가 필요하다.

---

# 9. 실제 블록체인의 최신 저장 구조

현재 주요 블록체인 구조는 대부분 다음과 같다.

### 구조

```
BlockChain
   │
   ├ Block
   │   ├ Header
   │   │   ├ stateRoot
   │   │   ├ txRoot
   │   │   └ receiptRoot
   │   │
   │   └ Transactions
   │
   └ State Database
        └ Merkle Patricia Trie
```

이 구조는 특히

Ethereum
에서 표준처럼 사용된다.

---

# 10. 최신 블록체인들이 바꾸는 방향

최근에는 상태 트리도 개선되고 있다.

대표 구조

### Verkle Tree

특징

* 훨씬 작은 증명
* 더 빠른 검증
* 더 작은 데이터

이 구조는

Ethereum
차세대 업그레이드로 연구되고 있다.

---

# 11. 결론 (중요)

블록체인은 **MongoDB 같은 테이블 기반 전체 저장 구조를 사용하지 않는다.**

현재 표준 구조는 다음과 같다.

```
Block
 ├ block header
 │   ├ state root
 │   ├ transaction root
 │   └ receipt root
 │
 ├ transaction list
 │
 └ state trie
```

실제 저장은

```
Key-Value DB
(LevelDB / RocksDB)
```

논리 구조는

```
Merkle Tree
Merkle Patricia Trie
```

이다.

---

1️⃣ **Ethereum 노드가 실제로 디스크에 데이터를 저장하는 구조 (폴더 구조)**
2️⃣ **State Trie가 어떻게 계정 데이터를 저장하는지**
3️⃣ **Solana / Aptos 같은 최신 블록체인의 저장 구조 차이**
4️⃣ **블록체인을 직접 만들 때 가장 효율적인 저장 아키텍처**
5️⃣ **MongoDB + 블록체인 혼합 구조 설계 방법**

이건 실제 **블록체인을 직접 만들 때 가장 중요한 핵심 구조**라서 제대로 이해하면 설계 수준이 완전히 달라진다.
