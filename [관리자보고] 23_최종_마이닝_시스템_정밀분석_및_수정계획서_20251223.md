### [추가 분석] 로그아웃 후 인증 실패 원인 (2025.12.23)
*   **현상:** 마이닝 페이지에서 2차 비밀번호 설정 후 잘 이용하다가, 로그아웃 후 재접속 시 "인증 실패" 발생.
*   **원인:** `WalletService.setSecondPassword()`가 비밀번호 해시를 **로컬(`localStorage`)에만 저장**하고, **서버에는 전송하지 않음**.
    *   로그아웃 시 보안 정책에 따라 `localStorage`가 삭제되므로, 유일하게 저장된 비밀번호 정보가 소실됨.
    *   재접속 시 서버(`getUserStatus`)에서 데이터를 불러오지만, 서버는 비밀번호 변경 사실을 모르므로 초기화 전/후의 불일치 데이터가 오거나 빈 값이 옴.
*   **해결책:** `WalletService.setSecondPassword` 메서드 수정. 비밀번호 설정 즉시 `apiService.registerUser`를 호출하여 **서버 DB에도 비밀번호 해시를 영구 저장**해야 함.

### [3차 심층 정밀 분석] (2025.12.23 20:30)
관리자 지시(세포 단위 분석)에 따른 코드 레벨 정밀 분석 결과입니다.

#### 1. 지갑 인증 무한 루프 및 2차 비번 오류 (Critical)
*   **증상:** 비밀번호가 설정된 상태에서도 [지갑 인증] -> [2차 비번 설정 모달]로 강제 이동하며 무한 루프.
*   **세포 단위 원인:**
    *   `WalletService.hasSecondPassword(address)` 메서드가 로컬 스토리지(`bw_wallet_data`)를 확인한 후, 없으면 **서버(`apiService.getUserStatus`)**에 물어봅니다.
    *   현재 백엔드(`server.js`)는 `secondPasswordHash`를 반환하지 않는 가짜 응답을 주거나 404 오류를 냅니다.
    *   결과적으로 `hasSecondPassword`는 실제 비번 설정 여부와 관계없이 `false`를 반환할 가능성이 높고, 이로 인해 `MiningAuthModal`은 "비번 미설정"으로 판단하여 설정 모달을 띄웁니다.
    *   2차적으로, 설정 모달에서 비번 설정을 시도하면 로컬 스토리지 충돌이나 서버 통신 실패로 에러가 발생합니다.
*   **해결책:** `hasSecondPassword`의 로직을 "로컬 스토리지 우선 & 서버 응답 정밀 검증"으로 수정하고, 모달에서 무조건 설정창으로 보내는 대신 "설정하시겠습니까?" 컨펌 창에서 [취소] 시 로그인 시도를 허용하는 탈출구를 만들어야 합니다.

#### 2. 추천 보너스 1BW 미지급 (Critical)
*   **증상:** 추천인과 신규 가입자 모두 보너스 보관함이 0BW.
*   **세포 단위 원인:**
    *   `WalletService.createWallet` 메서드에서 `referralReward`를 1로 설정하지만, 이는 단순 지갑 데이터일 뿐 **`ReferralBonusService`와 연동되지 않았습니다.**
    *   추천 보상 시스템의 핵심인 `ReferralBonusService.joinWithReferralCode()` 메서드가 지갑 생성 시점에 **호출되지 않고 있습니다.**
    *   따라서 `ReferralRecord`가 생성되지 않고, `rewardStorage`에 보상이 적립되지 않습니다.
*   **해결책:** `WalletService.createWallet` 성공 직후 `ReferralBonusService.joinWithReferralCode`를 호출하여 보상 시스템에 등록해야 합니다.

#### 3. 데이터 덮어쓰기 문제
*   **증상:** 새로고침 시 데이터 초기화.
*   **원인:** `ReferralBonusService.loadData`가 서버(`referrals.json`) 데이터를 우선시하여 로드하는데, 가짜 서버가 빈 데이터를 주면 로컬 데이터를 덮어써서 초기화시킵니다.
*   **해결책:** 서버 데이터가 유효한지(비어있지 않은지) 검증 후 로드하도록 방어 로직 강화.

### [4차 심층 원인 규명: 멀티 지갑 충돌과 서버의 한계] (2025.12.23 21:00)
관리자님께서 제보해주신 "다른 지갑 주소로 로그인 시 무한 루프" 현상과 "보너스 미지급"의 근본 원인을 파악했습니다.

#### 1. "1기기 1지갑" 구조적 한계 (The Single-Tenant Limit)
*   **현상:** 관리자님 지갑(A)이 로그인된 상태에서, 가입자 지갑(B)의 주소를 입력하고 [지갑 인증]을 시도하면 무한 루프에 빠집니다.
*   **원인:** 현재 `WalletService`는 보안을 위해 `localStorage`에 **단 하나의 지갑 데이터(`bw_wallet_data`)**만 저장합니다.
    *   로그인 시도 시, 저장된 지갑(A)과 입력된 주소(B)를 비교합니다.
    *   **다르기 때문에** 로컬 인증을 거부하고 서버에 문의합니다.
    *   하지만 **서버(`server.js`)는 가짜 서버라 B의 정보를 모릅니다.** (404 Not Found)
    *   결국 시스템은 "B는 신규 유저구나"라고 판단하여 [2차 비밀번호 설정]을 요구합니다.
    *   그러나 설정을 시도하면 `setSecondPassword` 내부에서 "저장된 지갑(A)과 입력된 주소(B)가 다르므로 보안상 접근 불가"라며 **설정을 조용히 거부(return false)**합니다. -> **무한 루프 발생.**

#### 2. 가짜 서버(Mock Server)의 데이터 휘발성
*   **현상:** 서버를 재시작하거나 로그아웃하면 추천 보너스 등이 사라짐.
*   **원인:** 현재 구동 중인 `server.js` 파일을 뜯어본 결과, **데이터를 파일에 저장하는 코드가 단 한 줄도 없습니다.**
    *   API 요청을 받으면 무조건 성공했다는 **가짜 응답(Fake Response)**만 보냅니다.
    *   프론트엔드는 "서버에 저장했다"고 믿고 있지만, 실제로는 허공에 외친 꼴입니다.
    *   이 때문에 `ReferralBonusService`가 서버에서 데이터를 불러오려 하면(새로고침 시), **빈 껍데기**만 받아오게 되어 로컬의 멀쩡한 데이터까지 오염시킬 위험이 있습니다.

#### 3. 최종 해결 제안 (The Definitive Fix)
이 모든 혼란을 잠재우기 위해 가장 확실한 방법은 **Mock Server(`server.js`)를 '진짜 파일 DB 서버'로 승격**시키는 것입니다.
1.  **`fs` 모듈 도입:** `users.json`, `referrals.json` 파일을 생성하여 데이터를 진짜로 쓰고 읽게 만듭니다.
2.  **API 구현:** 프론트엔드가 애타게 찾고 있는 `/api/user/register`, `/api/user/status`, `/api/referral/join` 등을 실제로 구현합니다.
3.  이렇게 하면, 지갑 A가 있는 PC에서 지갑 B로 로그인해도, 서버에서 B의 정보(암호화된 시드, 비번 해시)를 받아와서 **정상적인 "사용자 전환(Roaming)"이 가능**해집니다.

**이 서버 업그레이드 작업이 선행되지 않으면, 프론트엔드에서 아무리 코드를 고쳐도 "데이터 증발"과 "멀티 지갑 충돌"은 해결할 수 없습니다.**

### [6차 정밀 분석: 통계 UI 깨짐 및 데이터 연동 검증] (2025.12.24 10:25)
관리자님께서 지적하신 3가지 문제에 대한 정밀 재분석 결과 보고입니다.

#### 1. UI 깨짐 현상 (Overflow) [수정 시급]
*   **현상:** `21,000,000,000.00000000 BW` (약 25자)가 카드 영역을 벗어남.
*   **원인:** CSS가 긴 숫자에 대한 줄바꿈(Word Wrap)이나 폰트 크기 자동 조절을 처리하지 않음.
*   **해결 방안:** 관리자님 요청대로 **2단 분리 표기** 도입.
    ```
    21,000,000,000
    .00000000 BW
    ```
    *   정수부는 크게, 소수부는 작게 줄바꿈하여 배치하면 가독성과 UI 안정성을 동시에 확보 가능.

#### 2. 실시간 데이터 연동의 한계 (Simulation vs Real)
*   **분석:** 현재 서버(`server.js`)는 3초마다 자동으로 발행량(`totalReward`)을 증가시키는 **시뮬레이션 로직**을 사용 중입니다.
    *   사용자가 실제로 마이닝을 돌려서 얻은 보상이 서버로 전송되어 합산되는 구조가 **아닙니다.**
    *   따라서 개별 유저가 0.1개를 캐도, 전체 통계에는 반영되지 않고 서버는 자기 혼자 0.25개씩 늘어납니다.
*   **완벽 구현을 위한 요건:** 진정한 "실시간 연동"이 되려면, 사용자의 브라우저가 주기적으로(예: 1분마다) "나 이만큼 캤어"라고 서버에 보고(`Heartbeat`)하고, 서버가 이를 집계해야 합니다. 현재 구조는 이를 지원하지 않습니다.

#### 3. 지갑 생성 수 0 표시 문제 (Persistence)
*   **원인:** `server.js`의 `this.stats` 변수는 **메모리 변수**입니다.
    *   서버 코드를 수정하거나 재시작하면 이 변수는 초기화됩니다.
    *   실제 파일(DB)에 저장된 지갑 개수를 세어오는 로직(`fs.readdir`)이 없기 때문에, 서버가 켜질 때마다 0명(또는 하드코딩 초기값)으로 리셋됩니다.
*   **해결 방안:** 서버 시작 시 `fs` 모듈을 사용하여 `users.json` 파일 또는 데이터 폴더를 스캔하여 **실제 존재하는 지갑 파일 개수**로 `totalUsers`를 초기화해야 합니다.

## 2. [필수] 메인 홈페이지 실시간 데이터 연동 (Priority 1)
**목표:** 메인 화면의 채굴 현황판(6개 항목)이 살아있는 실제 데이터를 보여주도록 한다.

### 2-1. 실시간 통계 시스템 구축

*   **구현:**
    *   **백엔드:** 전체 가입자 수 및 총 채굴량을 집계하는 API (`/api/stats/global`) 최적화.

*   **지갑 생성 수:** 회원가입(지갑생성) 이벤트 발생 시 즉시 카운트 +1 반영.

## 3. [핵심] 네비게이션 메뉴별 실기능 개발 (Priority 2: 벤치마킹)
**목표:** 단순한 UI가 아닌, 레퍼런스 사이트(Pi Network, Coinness)와 동등한 수준의 **실제 기능 페이지**를 개발한다.

### 3-1. BW 블록 익스플로러 (Target: Pi BlockExplorer)
*   **기능 정의:**
    *   **실시간 블록 리스트:** 채굴 시스템에서 생성되는 블록 Height, 생성자, 보상 내역을 리스트 형태로 출력.
    *   **상세 조회:** 지갑 주소 또는 TXID 검색 시 해당 트랜잭션의 상세 내용을 보여주는 페이지 개발.
*   **개발 전략:** `BlockMonitorService`의 데이터를 시각화하는 전용 페이지(`ExplorerPage.tsx`) 신규 개발.

### 3-2. BW 커뮤니티 (Target: Coinness Community)
*   **기능 정의:**
    *   카테고리별 게시판 (공지사항, 자유게시판, Q&A).
    *   **CRUD 기능:** 글쓰기, 수정, 삭제, 목록 조회.
    *   **인터랙션:** 댓글 작성, 대댓글, 좋아요/싫어요 기능.
*   **개발 전략:** MongoDB `Community` 컬렉션 설계 및 백엔드 REST API 풀세트 구축 후 프론트엔드 연동.

### 3-3. BW 메인넷 & 노드 (Target: Pi Blockchain/Node)
*   **기능 정의:**
    *   BW 네트워크의 기술적 우수성을 설명하는 소개 페이지.
    *   노드 참여 방법 가이드 및 추후 노드 현황 모니터링 대시보드 공간 확보.
