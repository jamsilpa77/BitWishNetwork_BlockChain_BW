[기술 명세서] BitWish Network BW 채굴 시스템 보안 인증 구조 수정 및 UI 최적화

1. 개요 (Overview)

본 작업은 BitWish Network BW 포인트 채굴 시스템의 진입 장벽 역할을 하는 보안 인증 프로세스의 신뢰성 확보와 UI 가독성 향상을
목적으로 진행되었습니다.

기존 메인 홈 화면(HomePage.tsx)의 마이닝 시작 버튼 동작과 내부 보안 검증 모달(MiningAuthModal.tsx)의 레거시 로직
간에 발생하던 부조화를 해결하고, 브라우저 로컬 저장소(LocalStorage) 기반의 보안 지문(Fingerprint) 매칭 메커니즘을 견고히
재설정하였습니다.

3. 파일별 세부 수정 내역 (Modification Details)

3.1 HomePage.tsx (메인 채굴 상태 제어기)

  - 수정 부위: handleStartMining 함수 내부 분기 흐름 (Line 392 ~ 404 근처)
  - 세부 변경 사항:
      - 기존 구조는 walletService.getCurrentWalletAddress() 결과값이 유효하지 않을 경우,
        setIsWalletAuthModalOpen(true)를 호출하여 시드 구문 입력창으로 강제 유도하도록 처리되어 있었습니다.
      - 이를 개선하여 유효 주소의 반환 여부와 상관없이 항상 setIsMnemonicModalOpen(true)를 호출하여 6번
        화면(보안 인증 모달)이 무조건 먼저 열리도록 일원화하였습니다.
      - 수정 과정에서 발생할 수 있는 try-catch 문의 블록 누락 및 스코프 왜곡 문제를 구조적으로 정리하여, 컴파일 실패
        에러(return outside of function 및 TS1472)를 방지하였습니다.

3.2 MiningAuthModal.tsx (니모닉 파편 보안 인증 컴포넌트)

  - 수정 부위: 상태 변수(State), 검증 핸들러 및 JSX 마크업 구조
  - 세부 변경 사항:
      - 레거시 상태 제거: 6번 디자인 요구 사양에 맞추어 password 관련 로컬 상태 변수 및 유효성 검사 로직을 제거하여 불필요한
        렌더링 요소를 줄였습니다.
      - 로컬 보안 지문 검증 도입: 사용자가 입력한 주소 데이터와 브라우저 로컬 저장소에 격리 보관된 bw-wallet-address
        원본 데이터를 동적으로 상호 대조하는 handleVerifyAddress 함수를 구축하였습니다.
      - 지갑 미생성자 진입 차단: localStorage 내 등록 주소가 없을 시, 즉각 예외를 검출하고 상단 네비게이션을 통해 지갑을
        생성하도록 안내 문구를 표출하며 진입을 차단합니다.
      - TS 컴파일 경고 제어: 프로젝트의 tsconfig.json 검사 조건(noUnusedLocals)에 의해 미사용 컴포넌트
        프롭(onOpenSecondPassword)이 에러를 유발하지 않도록 useEffect 내부에서 명시적 더미 순환 참조
        처리를 수행하였습니다.

3.3 MiningAuthModal.css (보안 인증 모달 전용 스타일시트)

  - 수정 부위: 스타일시트 전면 재정리 및 확장 레이아웃 클래스 정의
  - 세부 변경 사항:
      - 주소 검증 입력 래퍼 추가: 입력 필드(auth-input)와 검증 단추(address-confirm-button)를 한 행으로
        직관적으로 나열할 수 있도록 display: flex; gap: 8px; 속성을 갖는
        .address-input-wrapper를 신설하였습니다.
      - 보안 안내 경고 박스 추가: 기획서 규격에 맞추어 경고 배경률(rgba(255, 193, 7, 0.1))과 경계선 및 가독성 높은
        노란색 텍스트 계열의 .warning-box 테마를 반영하였습니다.
      - 닫기 버튼 커스텀: 화면 하단의 복귀 경로인 .close-modal-btn에 차분한 색조 테마(#4a5568)를 입혀 핵심 트리거
        버튼과의 시각적 계층 구조를 명확히 나누었습니다.

4. 검증 메커니즘 흐름도 (Authentication Flow)

정상적인 인증 및 예외 처리는 아래와 같은 순서로 진행됩니다.

[마이닝 시작 클릭]
       │
       ▼
[보안 검증 창 오픈] (무조건 진입)
       │
       ├─► 입력 지갑 주소 ──► [로컬 저장소 조회]
       │                            │
       │                            ├─► 저장된 원본 주소 없음 (지갑 미개설 유저)
       │                            │     └─► "지갑 정보가 없습니다" 경고 출력 및 진입 차단 ❌
       │                            │
       │                            └─► 저장된 원본 주소 존재 (지갑 개설 완료 유저)
       │                                  │
       │                                  ├─► 입력 주소 !== 저장 주소
       │                                  │     └─► "주소가 일치하지 않습니다" 경고 출력 ❌
       │                                  │
       │                                  └─► 입력 주소 === 저장 주소
       │                                        └─► 10분 유예 하이패스 갱신 및 마이닝 페이지 진입 허가 ✅

5. 기대 효과 및 검토 의견 (Expected Effects)

  - 독립성 유지: BitWish Network 전용 가상 경제 채굴 아키텍처 규칙에 위배되는 전역 상태 도구 및 무거운 스텔라 네트워크 연동
    모듈을 사용하지 않고 오직 컴포넌트 자체 로컬 스토리지 대조만을 활용하여 완벽한 컴포넌트 독립성을 확보하였습니다.
  - 사용자 경험(UX) 개선: 사용자가 마이닝 과정에서 왜 지갑이 필요한지 명확히 인지하지 못한 채 시드 구문 입력창으로 곧장 넘어가는
    당혹스러운 흐름을 막고, 경고 메시지를 통한 자연스러운 학습적 인터페이스를 구현하였습니다.
