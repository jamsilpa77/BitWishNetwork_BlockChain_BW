BitWish 지갑 인터페이스 레이어 및 드래그 제어 기술 명세서 (BitWish Wallet Interface Layer & Drag
Control Technical Specification)

본 명세서는 BitWish 웹 프론트엔드 환경 내에서 발생하는 모달 레이어 간의 스택 계층(Stacking Context) 간섭, 마우스 포인터
이벤트 전달(Pointer Events Inheritance) 오류, 동적 중앙 정렬 수식 부재, 그리고 브라우저 의존적 에러 처리 등의
비즈니스 로직 및 레이아웃 한계를 해결하기 위해 적용된 기술적 변경 사항과 설계 방안을 기록한 사양 문서입니다.

1. 개요 (Overview)

지갑 관리 및 P2P 송·수신 환경의 사용자 경험(UX) 고도화를 목표로 삼았습니다. 컴포넌트 간의 레이어 계층 구조와 마우스 입력 신호의
흐름을 분석하여, 다음과 같은 세부 기능을 설계 및 수복하였습니다.

1.  쌓임 맥락(Stacking Context) 분석 및 Z-Index 계층 교정: 최상위 모달 위에 하위 제어 모달(송금, 수신, KYC
    가이드)이 가려지는 문제 해결.
2.  이벤트 흐름 제어(Pointer Events Control): 최상위 가림막 오버레이 환경에서 마우스 클릭 신호 유실 및 하단
    컴포넌트로의 포인터 오투과 문제 차단.
3.  기하 구조 기반 정중앙 좌표 매핑(Dynamic Centering): 부모 노드인 MyWalletModal의 위치에 맞게 하위 창이
    실시간으로 정중앙에 배치되는 상대 좌표 매칭 구현.
4.  마우스 이벤트 바인딩 드래그 메커니즘(Custom Drag UI): 고정 좌표계(fixed) 하에서 핸들 엘리먼트 드래그에 대응하는 고유
    마우스 조작 메커니즘 수립.
5.  인라인 에러 다이얼로그 수복(Inline Feedback System): 브라우저 기본 경고창(alert)을 배제하고 스타일 테마와
    조화되는 경고 레이아웃 탑재.
6.  미세 마진 보정(Layout Optimization): 겹침이 발생하는 입력 영역 하단 여백의 정확한 픽셀 단위 정량화.

2. 주요 설계 설계안 및 구현 상세

2.1. Z-Index 및 쌓임 맥락(Stacking Context) 교정

  - 원인: 브라우저는 HTML 요소들을 쌓을 때 일정한 '쌓임 맥락'을 따릅니다. 기존 구조는 다음과 같았습니다.
      - 부모 오버레이 및 모달: 10000 ~ 10100 수치 점유.
      - 하위 컴포넌트(TransferModal, KYCFormModal, premium-message-overlay): 고정형 10000
        수치 점유.
      - 이로 인해 새로운 제어 창이 활성화되어도 상위 지갑 모달에 가려 보이지 않거나 동작 불능 상태가 유도되었습니다.
  - 해결책:
      - 하위 모달의 오버레이 단위를 일괄 상향하여 10200 및 10250 대의 계층 구조를 신규 설계했습니다.
      - 수정된 적재 우선순위: premium-message-overlay (10250) > TransferModal &
        KYCFormModal (10200) > MyWalletModal (10100 이하)

2.2. 마우스 이벤트 포인터 투과(Pointer Events) 및 가림 제어

  - 원인: MyWalletModal은 화면 바탕의 타 컴포넌트 클릭을 위해 전체 백드롭에 pointerEvents: 'none'을
    적용했습니다. 이로 인해 자식 컴포넌트들까지 마우스 신호 차단 상태가 상속되었습니다. 결과적으로, KYC 창의 확인
    버튼이 클릭되지 않고, 뒷배경의 지갑 헤더만 클릭이 인식되어 손 모양 아이콘이 활성화되는 현상이 있었습니다.
  - 해결책:
      - 하위 팝업 레이어가 켜져 있는 상태(isKYCModalOpen || transferModal.isOpen ||
        messageModal.isOpen)를 실시간으로 감지하는 논리식을 부모 오버레이 스타일에 연동했습니다.
      - 팝업이 활성화되었을 때 부모 오버레이의 마우스 투과 상태를 즉시 해제(pointerEvents: 'auto')하고, 하위 팝업에도
        이를 명시하여 마우스 클릭 신호가 정확히 타겟 컴포넌트에 수렴되도록 보정하였습니다.

2.3. DOM 기하 구조 계산 기반 정중앙 좌표 추적 (Dynamic Centering)

  - 원인: 기기 화면 해상도의 편차 및 부모 모달인 MyWalletModal의 드래그 이동에 의해 지갑 창의 실시간 브라우저 좌표값은 상시
    변동됩니다. 하위 모달에 픽셀 상수를 그대로 주입하면 중앙에서 이탈하는 불안정한 레이아웃이 연출되었습니다.
  - 해결책:
      - TransferModal 활성화 시, 브라우저 가상 돔 API인
        document.querySelector('.my-wallet-modal')를 타겟팅하여 현재 지갑창의 영역
        정보(BoundingClientRect)를 획득합니다.
      - 지갑창의 현재 좌표(left, top)와 크기(width, height)를 추출하여 아래 수학 공식을 통해 동적으로 X, Y
        시작점을 갱신하게 하였습니다.
        \text{Target X} = \text{Parent Left} + \frac{(\text{Parent Width} - \text{Modal Width})}{2}
        \text{Target Y} = \text{Parent Top} + \frac{(\text{Parent Height} - \text{Modal Height})}{2}
      - 이로 인해 부모 지갑 창이 드래그되어 위치가 바뀐 상태여도, 송·수신 모달은 언제나 해당 부모 창 내부의 가로·세로 중앙 정점에
        위치합니다.

2.4. Independent Drag Logic (독립적 드래그 조작 제어)

  - 원인: 기존 TransferModal은 상대 정렬 중심의 흐름을 타서 좌표 위치 변경 및 마우스 이동량 계산이 구조적으로 불가능했습니다.
  - 해결책:
      - 컴포넌트 내부에 마우스 이동량을 연산하는 독립 상태값(position, isDragging, dragOffset)을 정립하고
        모달의 배치를 fixed 구조로 전환했습니다.
      - 특정 헤더 바 영역(.transfer-header-handle)에만 마우스 누름 이벤트(onMouseDown)를 부착하여 폼 내부
        인풋 컴포넌트 조작 시 원치 않게 창이 이동해 버리는 사용성 결함을 억제하고 정밀한 드래그 제어권을 마련했습니다.

2.5. 인라인 에러 다이얼로그 수복 (Inline Alert Layer)

  - 원인: 브라우저 기본 호스트 창(window.alert)을 호출할 경우 브라우저 렌더링 프레임 전체가 멈추고 서비스의 통일성을 훼손하는
    투박한 에러가 노출되었습니다.
  - 해결책:
      - 자체 에러 모달 상태값(alertModal)을 구성하고, 모달 최상단 노드 내부에 절대 정렬 영역(alertOverlay)과 경고
        바운더리 블록(alertBox)을 마운트했습니다.
      - 에러 발생 시 부모 모달 외부로의 조작이 자동 차단(이벤트 버블링 격리)되도록 설계하여 보안 대기 기간이나 인증 오류 상황에 대해
        일관성 있고 미려한 피드백을 출력합니다.

4. 파일별 영향도 명세 (Component Impact List)

4.1. MyWalletModal.tsx

  - 영향 범위: 지갑 전체 레이어를 감싸는 부모 컨테이너 및 락다운 경고 알림 모달.
  - 변경 제어 논리:
      - isKYCModalOpen || transferModal.isOpen || messageModal.isOpen 상태의 합집합
        유무에 맞춰 최외각 오버레이의 pointerEvents가 auto 혹은 none으로 조건부 변경되도록 설계하였습니다.
      - 안내 팝업의 마크업 내 zIndex를 10250으로 정립하고 pointerEvents: 'auto'를 인라인 스타일로 영구
        수복하여 이탈 없는 이벤트 통로를 복구했습니다.

4.2. TransferModal.tsx

  - 영향 범위: P2P 송금 및 수신을 제어하는 프레젠테이션 및 비즈니스 조작 모달.
  - 변경 제어 논리:
      - 컴포넌트의 가로 폭과 상태별 예상 세로 정량값을 기틀로 삼아 부모 컨테이너(.my-wallet-modal)의
        BoundingClientRect 값을 역추적하고, 마운트 직후 위치 값을 세밀하게 수정하는 훅 수식을 내장했습니다.
      - 기존의 차단된 에러 노출 방식을 전면 개편하여 setAlertModal 기반의 인터럽트 안내 뷰를 구현하고, 인라인 스타일 내역에
        alertOverlay, alertBox를 할당하였습니다.
      - 송금 수수료 텍스트(feeInfo)의 CSS 음수 마진 결함을 플러스 정밀 수치(4px)로 안전하게 교정하였습니다.

4.3. KYCFormModal.css

  - 영향 범위: 사용자 신원 인증 모달의 CSS 파일.
  - 변경 제어 논리:
      - .kyc-modal-overlay 클래스의 z-index를 지갑 창을 상회하는 10200으로 셋팅하였습니다.
      - pointer-events: auto; 속성을 클래스 선언자에 영구 바인딩하여 드래그 가림 현상 중 클릭 이벤트 통과 결함을
        완화하였습니다.

5. 결론 및 향후 유지보수 가이드 (Maintenance Guide)

본 설계 변경으로 인해 BitWish 지갑의 프론트엔드 모달 레이어 구조는 이벤트 흐름 제어 및 물리 좌표 제어 측면에서 한층 더 신뢰할 수
있는 기반을 가지게 되었습니다. 향후 컴포넌트 업데이트 시 하단 가이드를 참고하여 구조를 관리하는 것을 권장합니다.

1.  새로운 독립 모달(Portal) 추가 시 규칙: 지갑 시스템에 하위 모달을 새로 마운트할 때는 z-index 단위를 최소 10200
    이상으로 유지하고, 전체 가림막이 필요할 때에는 부모 오버레이의 마우스 투과 상태 제어 로직(isAnySubModalOpen과
    유사한 조건식)에 해당 상태값을 누락 없이 바인딩해야 합니다.
2.  드래그 영역 유지: 드래그가 필요한 팝업은 독립 구조의 일관성을 위해 반드시 fixed 속성과 마우스 리스너 추적 수식을 로컬 내부에
    내장하고, 헤더 영역에만 특정 제어 클래스(예: transfer-header-handle)를 적용하여 조작의 간섭을
    최소화하십시오.
