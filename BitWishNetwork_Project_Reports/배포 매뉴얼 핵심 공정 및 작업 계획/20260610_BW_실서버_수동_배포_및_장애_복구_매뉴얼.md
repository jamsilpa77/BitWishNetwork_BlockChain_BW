## 2. 수동 코드 배포 절차 (Manual Deployment Step-by-Step)

자동 배포 배치 파일이 동작하지 않는 경우, 아래 순서대로 명령어를 서버 터미널에 직접 입력하여 배포를 완료할 수 있습니다.

### [Step 1] 실서버 접속 및 최신 소스코드 수집 (Git Pull)
서버 터미널에 접속하여 프로젝트 최상위 디렉터리로 이동한 뒤, 깃허브 저장소로부터 최신 코드를 내려받습니다.
```bash
# 1. 앱 루트 경로로 이동
cd /root/app

# 2. 최신 코드를 git 저장소에서 당겨옴
git pull origin main
```

### [Step 2] 프론트엔드 Webpack 빌드 (UI 변경 시 필수)
사용자 화면(React)에 변경이 발생한 경우, 서버 내에서 아래 명령어로 직접 번들링(빌드) 작업을 수행합니다.
```bash
# 1. 마이닝 시스템 폴더로 이동
cd /root/app/BitWishNetwork_MiningSystem

# 2. 프로덕션 모드로 Webpack 빌드 실행 (약 2~3분 소요)
npx webpack --mode production --config webpack.config.js
```
* **빌드 결과물 확인:** 빌드가 정상적으로 완료되면 `dist/` 폴더 내에 `index.html` 및 압축/난독화가 완료된 `main.[hash].js`, `vendors.[hash].js`가 생성됩니다.

### [Step 3] 빌드 결과물 정적 배포 (Static Serve)
생성된 정적 파일들을 Nginx가 서빙할 수 있도록 `/var/www/html` 경로로 복사합니다.
```bash
# dist 내의 모든 웹 에셋을 웹 서버 루트 경로로 덮어쓰기 복사
cp -r /root/app/BitWishNetwork_MiningSystem/dist/* /var/www/html/
```

### [Step 4] PM2 백엔드 데몬 재구동 (API 로직 변경 시)
백엔드 로직(`server/` 하위 파일)에 대한 변경 사항이 적용되도록 PM2 백엔드 데몬을 재시작합니다.
```bash
# 1. 마이닝 시스템 폴더로 이동
cd /root/app/BitWishNetwork_MiningSystem

# 2. ts-node 프로젝트 설정을 지정하여 PM2 재시작 실행
TS_NODE_PROJECT=server/tsconfig.json pm2 restart bitwish-backend
```
* **상태 확인:** `pm2 status`를 입력하여 `bitwish-backend`가 `online` 상태로 구동 중이며 메모리를 안정적으로 할당받았는지 점검합니다.

## 3. 웹서버(Nginx) 설정 점검 및 긴급 수정

웹사이트가 `500` 또는 `502` 에러를 낼 경우, Nginx의 설정 파일을 점검하고 복구해야 합니다.

### [공정 1] Nginx 설정 파일 직접 편집
서버 내 Nginx 설정의 핵심 경로는 `/etc/nginx/sites-available/default` 입니다.
```bash
# 나노(nano) 에디터로 설정 파일 열기
nano /etc/nginx/sites-available/default
```

### [공정 2] Nginx 설정 파일 무결성 복구 본
아래 설정 구조를 복사하여 설정 파일에 그대로 붙여넣습니다. (주의: `$uri`, `$host` 등 앞에 백슬래시 `\`가 없어야 정상 동작합니다.)
```nginx
server {
    listen 80;
    server_name bitwishnetwork.com www.bitwishnetwork.com 141.164.40.101;
    return 301 https://bitwishnetwork.com$request_uri;
}

server {
    listen 443 ssl;
    server_name bitwishnetwork.com www.bitwishnetwork.com;

    ssl_certificate /etc/letsencrypt/live/bitwishnetwork.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/bitwishnetwork.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    # HTTP 보안 헤더 방어막
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

    # 민감 파일 차단 룰
    location ~ /\.(env|git|ssh|htaccess|htpasswd|bak|swp|log) {
        deny all;
        return 404;
    }

    location ~ /(tsconfig|package|package-lock|webpack\.config)\. {
        deny all;
        return 404;
    }

    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:5001/api/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### [공정 3] 설정 테스트 및 Nginx 재시작
파일을 저장한 뒤(Nano 기준: `Ctrl+O` -> `Enter` -> `Ctrl+X`), 설정의 문법적 무결성을 검사하고 서비스를 재기동합니다.
```bash
# Nginx 설정 파일 문법 검증 (반드시 Successful 메시지 확인 후 리스타트 진행)
nginx -t

# Nginx 서비스 재시작
systemctl restart nginx
```

## 4. 방화벽(UFW) 및 보안 서비스 긴급 복구

포트가 뚫리거나 무단 공격으로 CPU 부하가 걸리는 등의 상황 발생 시 조치 사항입니다.

### [1] 방화벽 포트 통제 규칙
외부에서 API 서버 포트(5001)에 직접 붙는 패킷을 차단하여 오버헤드와 해킹 시도를 물리적으로 방어합니다.
```bash
# 5001 백엔드 포트 외부 유입 거부
ufw deny 5001/tcp

# 4001 디바이스 연동 포트 외부 유입 거부
ufw deny 4001/tcp

# 웹 포트 개방 상태 확인 (80, 443, 22는 ALLOW 상태여야 함)
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow 22/tcp

# 방화벽 재시작
ufw reload

# 방화벽 현재 상태 확인
ufw status verbose
```

### [2] 침입 방지 시스템(Fail2ban) 점검
비정상 접속자를 밴(Ban) 처리하는 데몬의 생존 상태를 확인합니다.
```bash
# Fail2ban 가동 상태 확인
systemctl status fail2ban

# 서비스 오작동 시 재기동
systemctl restart fail2ban
```

## 5. 긴급 장애 대처 (Trouble Shooting)

### **Q1. 사이트에 "500 Internal Server Error"가 발생하는 경우**
* **해결법:** `/var/log/nginx/error.log` 파일의 끝단을 열어 rewrite loop가 생겼는지 점검합니다. Nginx 설정 파일 내에 `$uri` 등이 백슬래시와 함께 `\$uri`로 오발생되어 있는지 확인하고 위의 Nginx 무결성 설정 구조로 복구합니다.

### **Q2. 사이트에 "502 Bad Gateway"가 발생하는 경우**
* **해결법:** 백엔드 API 데몬(`bitwish-backend`)이 정지된 상황입니다. `pm2 status`로 서버 상태 확인 후, 꺼져 있다면 `pm2 start bitwish-backend` 또는 `pm2 restart bitwish-backend`를 입력하여 백엔드를 가동해 주십시오.

### **Q3. DB 정합성 문제 또는 장부 불일치 오류가 발생하는 경우**
* **해결법:** 백엔드 재기동 후 `/root/.pm2/logs/bitwish-backend-out.log`를 열어 `[수복 엔진]` 로그를 모니터링합니다. 수복 엔진이 30초마다 돌며 유저의 누적 상태를 복원하고 체인 블록 불일치를 자동으로 정정(`autoRestoreMiningStates`)하므로, 약 1~2분 정도 지켜보시면 정상 상태로 원상 복구됩니다.
