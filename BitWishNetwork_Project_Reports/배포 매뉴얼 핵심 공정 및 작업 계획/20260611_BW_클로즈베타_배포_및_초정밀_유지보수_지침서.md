# 📘 BitWish Network 클로즈베타 배포 및 초정밀 유지보수 지침서

본 지침서는 **BitWish Network** 마이닝 시스템 및 블록체인 코어의 개발, 배포, 유지보수를 담당할 후임 개발자 및 AI 에이전트를 위한 **초정밀 가이드라인**입니다. 

컴퓨터 재설치, 에이전트 교체, 혹은 소통 단절 상황이 발생하더라도 본 지침서의 규격과 순서를 그대로 준수하면 오류나 데이터 깨짐 없이 안전하게 서비스를 업데이트하고 상용 서버를 제어할 수 있습니다.

## 2. ⚠️ 개발 시 인코딩 및 코드 변경 절대 수칙 (AI 필수 숙지)

새로운 AI 에이전트나 작업자가 소스 코드를 편집할 때 **반드시 지켜야 할 철칙**입니다. 이를 위반할 경우 심각한 한글 깨짐이나 통신 장애가 발생합니다.

### 🚫 [수칙 1] 시스템 쉘(PowerShell/Bash)을 이용한 파일 쓰기 금지
*   Windows PowerShell의 `Set-Content` 또는 `>` 리디렉션을 사용해 코드를 쓰면 파일 인코딩이 **ANSI 또는 UTF-16**으로 강제 변경되어 **소스 내의 모든 한글 주석과 다국어 텍스트가 깨집니다.**
*   **해결책**: 모든 코드 변경은 개발 도구(IDE)의 전용 파일 편집 API를 통해서만 수행해야 합니다.

### 🚫 [수칙 2] 실서버 파일 직접 수정 금지
*   실서버인 `/root/app` 경로 내부에서 소스코드를 직접 수정하거나 편집하지 마십시오.
*   **해결책**: 모든 기능 수정은 **로컬 워크스페이스에서 작업 -> Git 커밋 및 Push -> 배포 스크립트를 통한 실서버 빌드** 프로세스를 준수해야 합니다.

### 🚫 [수칙 3] API 통신 주소(`API_BASE_URL`) 하드코딩 금지
*   `BitWishNetwork_MiningSystem/src/services/ApiService.ts` 파일의 API 통신 주소는 절대 특정 IP나 `http://localhost:5001/api`로 고정해서는 안 됩니다.
*   반드시 아래와 같이 사용자의 접속 도메인에 따라 동적으로 가리키도록 설정해야 실서버 배포 시 통신 장애가 나지 않습니다.
    ```typescript
    let API_BASE_URL = '/api'; // 실서버에서는 동일 도메인 상대 경로를 이용해 Nginx 리버스 프록시 통과
    try {
        if (typeof process !== 'undefined' && process.env && process.env.REACT_APP_API_URL) {
            API_BASE_URL = process.env.REACT_APP_API_URL;
        } else if (typeof window !== 'undefined' && window.location.hostname === 'localhost') {
            API_BASE_URL = 'http://localhost:5001/api'; // 로컬 개발 시에만 5001 포트 사용
        }
    } catch (error) {
        console.warn('Failed to load environment variable, using default URL');
    }
    ```

## 3. 🛠️ 배포 전 로컬 빌드 및 사전 검증 절차

코드를 커밋하고 배포 서버에 반영하기 전, 문법 오류로 인한 빌드 중단을 예방하기 위해 로컬 검증을 거칩니다.

1.  **로컬 의존성 설치 (필요시)**
    ```powershell
    cd c:\BitWishNetwork_BlockChainMainnet\BitWishNetwork_MiningSystem
    npm install --legacy-peer-deps
    ```
2.  **로컬 웹팩 빌드 테스트**
    ```powershell
    npm run build
    ```
    *   `webpack.config.js`가 구동되어 `dist/` 폴더에 번들이 정상적으로 추출되는지 확인합니다. 빌드 경고가 아닌 **에러(Error)**가 없어야 배포 준비가 완료된 것입니다.

## 4. 🚀 실서버 배포 파이프라인 (Step-by-Step)

### 1) 자동 배포 방식 (권장)
로컬 프로젝트 루트 디렉터리(`c:\BitWishNetwork_BlockChainMainnet`)에서 배포 스크립트를 가동합니다.
```powershell
# PowerShell 창에서 실행
.\deploy.bat
```
*   **자동 처리되는 공정:**
    1.  로컬 변경 사항을 자동으로 스테이징 및 커밋 생성.
    2.  원격 깃허브 저장소(`origin main`)로 코드 푸시.
    3.  로컬 SSH 에이전트를 가동해 실서버(`141.164.40.101`)에 원격 접속.
    4.  실서버에서 `git pull` 실행.
    5.  프론트엔드 수정사항(`src/` 폴더 내) 감지 시 자동으로 웹팩 프로덕션 빌드 후 정적 경로 복사.
    6.  PM2 백엔드 프로세스(`bitwish-backend`) 재부팅.

---

### 2) 수동 배포 방식 (배포 도구 에러 시 대처법)
자동 배포 스크립트에 문제가 생기거나 SSH 인증 라이브러리 충돌 시, 직접 Git과 서버 명령어로 배포하는 공정입니다.

#### [로컬 PC 작업]
```powershell
# 1. 로컬 루트 디렉터리로 이동
cd c:\BitWishNetwork_BlockChainMainnet

# 2. 로컬 수정사항을 커밋 후 원격 저장소에 반영
git add .
git commit -m "Auto-deploy update: [변경 사항 설명]"
git push origin main
```

#### [실서버 SSH 접속 후 작업]
```bash
# 1. SSH 키를 이용해 서버에 접속
ssh root@141.164.40.101

# 2. 실서버 앱 경로로 이동하여 최신 소스 수집
cd /root/app
git pull origin main

# 3. 프론트엔드 빌드 및 정적 경로 배포 (프론트 변경 시에만 실행)
cd /root/app/BitWishNetwork_MiningSystem
npx webpack --mode production --config webpack.config.js
cp -r /root/app/BitWishNetwork_MiningSystem/dist/* /var/www/html/

# 4. 백엔드 PM2 데몬 재기동
cd /root/app/BitWishNetwork_MiningSystem
TS_NODE_PROJECT=server/tsconfig.json pm2 restart bitwish-backend

# 5. 실행 상태 최종 검증
pm2 status
```

## 5. ⚙️ 서버 인프라 환경설정 사양서

### 1) Nginx 리버스 프록시 및 보안 설정
설정 파일 경로: `/etc/nginx/sites-available/default`
*   사용자가 `http`로 접속 시 무조건 `https`로 리디렉션 처리합니다.
*   `/api/` 경로로 들어오는 모든 요청은 로컬 백엔드 포트(`http://127.0.0.1:5001/api/`)로 프록시 패스합니다.

```nginx
server {
    listen 80;
    server_name bitwishnetwork.com www.bitwishnetwork.com 141.164.40.101;
    return 301 https://bitwishnetwork.com$request_uri;
}

server {
    listen 443 ssl;
    server_name bitwishnetwork.com www.bitwishnetwork.com;

    ssl_certificate /etc/letsencrypt/live/bitwishnetwork.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/bitwishnetwork.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    # [보안] HTTP 보안 헤더 설정
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

    # [보안] 민감 경로 및 소스코드 직접 접근 차단
    location ~ /\.(env|git|ssh|htaccess|htpasswd|bak|swp|log) {
        deny all;
        return 404;
    }
    location ~ /(tsconfig|package|package-lock|webpack\.config)\. {
        deny all;
        return 404;
    }

    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:5001/api/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
*   설정 파일 수정 후 반드시 `nginx -t`로 검증한 뒤 `systemctl restart nginx`를 통해 재동작시킵니다.

### 2) 백엔드 데몬 제어 (PM2)
*   **데몬 기동 명령어**:
    ```bash
    TS_NODE_PROJECT=server/tsconfig.json pm2 start server/index.ts --name "bitwish-backend"
    ```
*   **프로세스 상태 모니터링**:
    ```bash
    pm2 status                # 구동중인 프로세스 리스트
    pm2 logs bitwish-backend  # 실시간 로그 스트리밍
    ```

### 3) 방화벽(UFW) 통제 규칙
포트 외부 노출을 제한하여 해킹 및 디도스 시도를 원천 차단합니다.
*   **포트 통제 구성:**
    *   `TCP 80` (HTTP): 개방 (`allow`)
    *   `TCP 443` (HTTPS): 개방 (`allow`)
    *   `TCP 22` (SSH): 개방 (`allow`)
    *   `TCP 5001` (백엔드 포트): **차단 (`deny`)** -> 반드시 Nginx 프록시를 통해서만 접근 가능해야 합니다.

## 6. 🚨 긴급 장애 대처 (Trouble Shooting)

### 1) "502 Bad Gateway" 에러 발생 시
*   **원인**: Node.js 백엔드 데몬이 다운되었거나 작동하지 않는 경우입니다.
*   **대처법**: SSH 접속 후 `pm2 status`를 확인하고, 꺼져 있다면 아래 명령으로 복구합니다.
    ```bash
    cd /root/app/BitWishNetwork_MiningSystem
    TS_NODE_PROJECT=server/tsconfig.json pm2 restart bitwish-backend
    ```

### 2) "500 Internal Server Error" 에러 발생 시
*   **원인**: Nginx의 라우팅 설정 문법 오류 또는 백엔드 내부의 핸들되지 않은 심각한 예외 상황입니다.
*   **대처법**:
    1.  `/var/log/nginx/error.log`를 확인하여 Nginx 설정 내 변수(예: `$uri`의 오탈자 여부)를 검증합니다.
    2.  `pm2 logs bitwish-backend`를 통해 백엔드 런타임 오류 로그를 확인합니다.

### 3) 채굴 데이터 유실 및 블록체인 노드 수동 수복이 필요할 때
*   **원인**: 서버 불시 중단 또는 네트워크 장애로 인해 유저들의 로컬 채굴 데이터와 블록체인 데이터베이스 간 불일치가 생긴 경우입니다.
*   **대처법**: 백엔드 서버가 켜질 때 자동으로 데이터 무결점 대조 및 복원 수복 엔진(`autoRestoreMiningStates`)이 최초 1회 실행되고, 이후 30초마다 계속 동기화 작업을 수행합니다. 따라서 **백엔드를 단순히 1회 재시작하고 1~2분 정도 대기**하면 누락된 블록이 자동으로 재생성 및 매핑되며 장부가 안전하게 수복됩니다.
