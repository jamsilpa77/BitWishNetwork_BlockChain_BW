## 1. 정책 지시서 핵심 요구사항
*   **"추천 보너스(2% 채굴분)와 추천 보상(1BW 가입보상)은 각각 별도의 보관함에 영구 저장 및 표시되어야 한다."**
*   **"지갑은 어떠한 상황(새로고침, 기기 변경 등)에서도 정확한 보관함 잔액을 표시해야 한다."**

## 2. 이전 문제점 (Previous Faults)
*   **[백엔드 데이터 누락]** 프론트엔드가 사용자 정보를 요청(`/mining/status`)할 때, 백엔드는 `User` 테이블만 조회하고 **`BonusRecord` 테이블은 조회하지 않음.**
    *   **결과:** 프론트엔드는 보너스 정보를 전혀 받지 못해 무조건 **0**으로 표시됨. (정책 위반: 데이터 영구 보존 실패)
*   **[프론트엔드 불안정성]** `MyWalletModal`이 `localStorage`에 의존하여, 브라우저 캐시 삭제 시 데이터가 즉시 증발함.
*   **[UI 텍스트 혼란]** "추천 보상", "보너스 보관함" 등 용어가 통일되지 않아, 마이닝 페이지와 불일치를 초래함.

## 3. 수정 및 구현 내역 (Current Implementation)

### ✅ Step 2-1. [Server] 데이터 조회 로직 강화
*   **파일:** `server/controllers/MiningController.ts`
*   **내용:** `getUserStatus` 메서드 내부에서 **`BonusRecord` 모델을 명시적으로 조회(Join)**.
*   **코드:**
    ```typescript
    const bonusRecord = await BonusRecord.findOne({ walletAddress });
    // ...
    user: {
        ...user.toObject(),
        referralBonusStorage: bonusRecord?.referralBonusStorage || '0',  // 2% 채굴분
        referralRewardStorage: bonusRecord?.referralRewardStorage || '0', // 1BW 가입보상
        referralList: bonusRecord?.referralList || []
    }
    ```
    *   이제 백엔드는 클라이언트 요청 시 **항상 최신 보너스 잔액**을 포함하여 응답합니다.

### ✅ Step 2-2. [Client] 데이터 연동 및 UI 표준화
*   **파일:** `src/components/MyWalletModal/MyWalletModal.tsx`
*   **내용 1 (데이터 연동):** `localStorage` 의존성을 끊고, `apiService.getUserStatus`를 직접 호출하여 **서버 DB 데이터**를 화면에 표시. (Anywhere Access 기반 마련)
*   **내용 2 (UI 텍스트):** 마이닝 페이지와 100% 동일하게 레이블 수정.
    *   **"추천 보너스 보관함"** (2% 채굴분)
    *   **"추천 보상 보관함"** (1BW 가입보상)

## 4. 시스템 작동 원리 및 향후 효과
1.  **순환 구조 완성:** 사용자가 추천인 가입(Step 5 예정) -> DB(`BonusRecord`)에 저장 -> 지갑 열기(Step 2 완료) -> `MiningController` 조회 -> **지갑에 정확한 잔액 표시.**
2.  **데이터 무결성:** `localStorage`를 쓰지 않으므로, 사용자가 브라우저를 바꿔도, 캐시를 지워도 **내 자산 정보는 안전하게 표시**됩니다.
3.  **정책 준수:** 지시서가 요구한 "별도 보관함"과 "정확한 표시"를 완벽하게 충족했습니다.

## 5. 결론
Step 2 작업은 **"정확한 데이터를 읽어오는 파이프라인(Read Pipeline)"의 완성**입니다.
이제 이 파이프라인에 **실제 데이터를 기록(Write)하는 Step 5(추천인 저장 API)** 작업이 수행되면, 전체 추천 시스템은 완벽하게 작동하게 됩니다.
