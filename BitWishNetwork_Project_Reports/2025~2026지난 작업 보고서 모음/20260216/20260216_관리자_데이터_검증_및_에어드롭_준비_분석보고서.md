# [분석] 관리자 데이터 검증 및 1:1 에어드롭 준비 분석 보고서

**작성일:** 2026년 2월 16일
**작성자:** BitWish AI Assistant
**목표:** 향후 진행될 KYC 기반 1:1 토큰 에어드롭(Migration)을 위해, 현재 시스템이 **필수 사용자 데이터(가입일, 채굴량, 추천관계 등)**를 완벽하게 기록하고 조회할 수 있는지 정밀 진단함.

---

## 1. 개요 (Background)
BitWish Network는 추후 블록체인 메인넷 런칭 시, 현재 마이닝 시스템에서 채굴된 포인트를 **실제 토큰으로 1:1 교환(Airdrop)** 해주는 정책을 가지고 있다. 이를 위해서는 시스템이 **"누가, 언제 가입해서, 얼마를 채굴했고, 누구를 추천했는지"**에 대한 **위변조 불가능한 원장(Ledger) 데이터**를 보유하고 있어야 하며, 관리자는 이 모든 데이터를 **투명하게 조회하고 검증**할 수 있어야 한다.

---

## 2. 현재 관리자 시스템 정밀 진단 (Current Status Analysis)

### 2-1. 데이터 보유 현황 (DB Level)
다행히 MongoDB 모델 상으로는 핵심 데이터가 저장되고 있음이 확인되었다.

| 데이터 항목 | 저장 위치 (Collection.Field) | 상태 | 비고 |
| :--- | :--- | :--- | :--- |
| **지갑 주소** | `User.walletAddress` | ✅ 정상 | 유일성 보장됨 |
| **가입 일시** | `User.createdAt` | ✅ 정상 | `timestamps: true` 옵션으로 자동 기록됨 (년월일시분초) |
| **추천 코드** | `User.myReferralCode` | ⚠️ **누락 가능성** | 일부 로직 미비로 지갑 생성 시 생성 안 되는 버그 존재 (수정 필요) |
| **추천인 관계** | `User.referrerCode` | ✅ 정상 | 가입 시 기입한 코드 저장됨 |
| **채굴 누적량** | `MiningState.accumulatedReward` | ✅ 정상 | 실시간 채굴량 기록 중 |
| **보너스 보관함** | `BonusRecord.referralBonusStorage` | ⚠️ **필드 혼재** | 과거 `bonusStorage`와 현재 필드 혼용으로 집계 오류 발생 (수정 필수) |
| **KYC 상태** | `User.isKycVerified` | ✅ 정상 | 기본값 false, 추후 인증 시 true 변경 가능 |
| **2차 비번/니모닉** | `User.secondPasswordHash` 등 | ✅ 저장됨 | **[보안]** 해시/암호화되어 관리자도 원문 조회 불가능 (정상) |

### 2-2. 관리자 조회 기능 현황 (Admin Page Level)
데이터는 있으나, 관리자가 이를 **조회하고 검증하는 도구(API/UI)**가 매우 부족하거나 잘못 구현되어 있다.

1.  **전체 지갑 목록 조회 기능 부재:**
    *   현재 관리자는 특정 지갑 주소를 알아야만 검색이 가능함 (`/mining/:address`).
    *   **문제점:** 전체 가입자가 10만 명이라면, 관리자는 누가 가입했는지 목록을 볼 수가 없어 **전수 조사가 불가능함.**
2.  **가입 일시(Time) 디테일 부족:**
    *   현재 통계 API는 단순 카운트만 보여줌.
    *   **필요 기능:** "2026-02-16 10:00:00 ~ 11:00:00 가입자 목록" 처럼 **초 단위 정밀 조회**가 가능해야 함.
3.  **보상 데이터 신뢰성 결여 (Issue 1):**
    *   앞서 발견된 "6 BW 유령 데이터"처럼, 리셋 후에도 데이터가 남는 등 **표기 오류**가 있어 에어드롭 시 과지급/미지급 사고 위험이 큼.

## 3. 에어드롭을 위한 필수 데이터 조회/검증 요건 (Requirements)

향후 1:1 교환 시 분쟁을 예방하고 정확한 지급을 위해, 다음 기능이 **관리자 페이지에 반드시 구현**되어야 한다.

### 3-1. [통합 원장 조회] 기능 (Master Ledger View)
*   **기능:** 모든 가입자의 핵심 데이터를 한눈에 엑셀처럼 조회/다운로드 가능해야 함.
*   **필수 포함 컬럼:**
    1.  **NO:** 연번
    2.  **Wallet Address:** 지갑 주소 (고유 식별자)
    3.  **Joined Date:** 가입 년/월/일 시:분:초 (정밀 타임스탬프)
    4.  **Referral Code:** 본인 코드 (관계 증명용)
    5.  **Referrer Code:** 추천인 코드 (보너스 출처 증명용)
    6.  **Mining Reward:** 순수 채굴량 (BW)
    7.  **Referral Bonus:** 추천 보너스 총합 (BW)
    8.  **Total Asset:** 총 자산 (Mining + Bonus)
    9.  **Real Status:** 채굴 중/중지, KYC 여부

### 3-2. [데이터 무결성 검증] 도구 (Integrity Check)
*   **기능:** DB 내 "총 발행량"과 "유저별 보유량 합계"가 일치하는지 실시간 대조.
*   **목적:** "6 BW 유령 데이터" 같은 찌꺼기를 즉시 적발하여 0원 오차(Zero Error)를 보장해야 함.

## 4. 결론 및 제안
현재 시스템은 데이터는 쌓고 있으나, 이를 **검증하고 추출할 수단**이 전무하다.
이대로라면 에어드롭 시점에 "누가 언제 가입했는지", "이 코인이 진짜인지" 증명할 길이 없어 대형 사고로 이어질 수 있다.

따라서 작업을 중단하는 것이 아니라, **오히려 관리자 페이지의 "전체 지갑 통합 조회(List All Users)" 기능 개발**을 최우선 과제로 삼아야 한다.
이는 단순 편의 기능이 아니라, **블록체인 사업의 신뢰성(Auditability)을 위한 필수 인프라**이다.
