## 1. 개요 및 긴급 진단 결과
현재 [나의 지갑]에서 [마이닝 시작] 버튼 클릭 시, 단순히 화면만 전환되고 실제 데이터 연결 고리(`walletAddress`)가 끊겨 **"Not Connected"** 상태가 되는 현상이 확인되었습니다.

단순히 주소값만 넘겨준다고 해결되는 것이 아니며, **인증 세션(Auth Session) 생성**과 **데이터 초기화 시점(Initialization Timing)**이 맞지 않으면 "주소는 있는데 로그인은 안 된" 기형적인 상태가 될 위험이 있습니다.

## 2. 정밀 분석 (Deep Dive Analysis)

### 2-1. 현재 데이터 흐름의 끊김 (Broken Link)
*   **발신지 (`MyWalletModal`):** 버튼 클릭 시 `onOpenMining`을 호출하지만, 주소(`walletAddress`)를 인자로 실어 보내지 않음. (현재 수정 대기 중)
*   **중계지 (`HomePage`):** `onOpenMining` 호출 시 `setAuthenticatedAddress`를 실행하지 않고 바로 모달만 엶. -> **데이터 공백 발생**
*   **수신지 (`MiningStatusModal`):** `walletAddress` props가 비어있으므로 `useEffect` 내부의 `initializeData`가 실행되지 않음. -> **"Not Connected" 출력**

### 2-2. 잠재적 위험 요소 (Hidden Risks)
1.  **인증 세션 부재:** `MyWalletModal`을 통해 들어오면 `MiningAuthModal`(로그인 창)을 건너뛰게 되는데(Free Pass), 이때 `bw_mining_auth` 로컬 스토리지 세션이 생성되지 않을 수 있습니다. 
    *   **위험:** 새로고침 하면 로그인이 풀려버림.
    *   **해결책:** `HomePage`에서 프리패스 처리 시 강제로 세션을 생성해야 함.

2.  **데이터 로딩 레이스 컨디션:** `authenticatedAddress` 상태 업데이트(`setState`)는 비동기인데, 모달이 열리는 속도가 더 빠르면 `null` 값을 먼저 읽을 수 있음.
    *   **해결책:** `setAuthenticatedAddress`와 세션 생성을 먼저 수행하고 모달을 열도록 보장해야 함.

## 3. 완전 해결 솔루션 (Complete Solution Architecture)

단순 땜질이 아닌, **"인증 - 데이터 전송 - 세션 유지"** 3박자가 맞는 완벽한 흐름을 구축합니다.

### **[Step 1] `MyWalletModal.tsx` (발신자)**
*   **역할:** 정확한 주소 전달.
*   **코드 변경:** (이전 계획과 동일)
    ```typescript
    onOpenMining(walletAddress); // 1. 주소를 실어 보냄
    ```

### **[Step 2] `HomePage.tsx` (중계자 & 인증 부여)**
*   **역할:** 주소 수령 + **세션 생성(핵심)** + 모달 오픈.
*   **코드 변경:**
    ```typescript
    onOpenMining={(address) => {
        // 1. 상태 업데이트
        setAuthenticatedAddress(address);
        
        // 2. [중요] 강제 세션 생성 (새로고침 해도 유지되도록)
        // MiningAuthModal을 안 거치므로 여기서 수동으로 찍어줘야 함.
        const sessionData = {
            address: address, // 주소
            timestamp: Date.now(),
            method: 'WALLET_DIRECT_ACCESS' // 경로 추적용
        };
        localStorage.setItem('bw_mining_auth', JSON.stringify(sessionData));
        
        // 3. 모달 오픈
        setIsMyWalletModalOpen(false);
        setIsMiningModalOpen(true);
    }}
    ```

### **[Step 3] `MiningStatusModal.tsx` (수신자)**
*   **역할:** 데이터 무결성 검증.
*   **검증:** 이미 로직(`useEffect`의 `if (!walletAddress) return;`)은 튼튼하므로, 위에서 주소만 잘 들어오면 자동으로 초기화됨. 별도 수정 불필요.

## 5. 추가 정밀 검증: 정식 루트와의 동일성 보장 (Identity Verification)

### 5-1. 문제 제기
'나의 지갑'에서 [마이닝 시작]을 눌러 진입하는 것(Free Pass)이, 메인 화면에서 [마이닝 & 보너스] 버튼을 눌러 정식으로 2차 비밀번호를 입력하고 진입하는 것과 **완벽하게 동일한 권한과 상태**를 가지는가?

### 5-2. 데이터 구조 비교 및 일치화 전략
정식 루트(`MiningAuthModal`)를 통과할 때 생성되는 데이터와 프리패스 데이터를 1:1로 매핑하여 차이가 없도록 한다.

| 구분 | 정식 루트 (MiningAuthModal) | 프리패스 루트 (MyWalletModal) | 일치화 방안 |
| :--- | :--- | :--- | :--- |
| **인증 주체** | `MiningAuthModal` | `HomePage` (MyWalletModal의 위임) | `HomePage`가 `MiningAuthModal`의 역할을 대행함 |
| **세션 키** | `bw_mining_auth` | `bw_mining_auth` | **동일** (키 이름 일치 필수) |
| **세션 값** | `{ address: 'BW...', timestamp: 123... }` | `{ address: 'BW...', timestamp: 123... }` | **동일** (JSON 구조 일치 필수) |
| **메모리 상태** | `authenticatedAddress` (State) | `authenticatedAddress` (State) | `setAuthenticatedAddress`로 동기화 필수 |
| **2차 비번 검증** | 실시간 입력값으로 검증함 | **이미 '나의 지갑' 진입 시 검증됨** | **신뢰의 전이(Trust Transfer)**: 나의 지갑에 들어왔다는 것 자체가 인증된 상태이므로, 시스템은 이를 '검증 완료'로 간주해야 함. |

### 5-3. 구현 시 필수 준수 사항 (Critical Implementation Detail)
`HomePage.tsx`에서 세션을 생성할 때, 단순 문자열이 아닌 **시스템이 인정하는 정규 포맷(JSON)**으로 정확히 박아줘야 한다.

```typescript
// HomePage.tsx 수정 시 적용할 코드 예시
const sessionData = {
    address: address, // 주소
    timestamp: Date.now(),
    method: 'WALLET_DIRECT_ACCESS' // 경로 추적용
};
localStorage.setItem('bw_mining_auth', JSON.stringify(sessionData));
```

이 과정을 거치면, 마이닝 페이지(`MiningStatusModal`)는 유저가 어디서 왔는지 구분할 필요 없이 **완벽하게 동일한 정식 유저**로 인식하게 된다.

**최종 결론:** 위 로직을 적용하면, **'나의 지갑 프리패스' = '2차 비번 정식 통과'**가 시스템적으로 **100% 동일성**을 갖게 된다.
