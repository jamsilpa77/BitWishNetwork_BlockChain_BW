## 1. [문제 근본 분석] 코드는 왜 바뀌었는가?

### **(1) 발행 기관(로직)의 변질**
- **작업 위치:** `src/services/BlockchainService/WalletService.ts`
- **현상:** `generateReferralCode` 함수의 수식이 어느 시점에 **"지갑 주소의 끝 7자리"**를 가져오도록 개악되었습니다.
- **증거:** 
    - **정상 코드:** `REF9F5FF0909DC5` (총 15자리)
    - **변질 코드:** `REF9F5FF090B44BFA8` (총 18자리)
    - **분석:** 현재 로직은 지갑 주소(`BW9F5FF0...BFA8`)의 끝에서 **`B44BFA8` (7자리)**를 떼와서 붙입니다. 사용자님이 말씀하신 `9DC5`는 과거의 다른 공식(혹은 4자리 추출)에 의한 결과물입니다.

### **(2) 자동 오염 발생**
- 사용자가 지갑을 열 때마다 위 '개악된 로직'이 본인의 지갑 주소를 보고 코드를 새로 계산합니다.
- 계산된 '변질 코드'를 서버에 전달하고, 서버는 DB에 있던 '정상 코드' 위에 이 쓰레기 데이터를 덮어써 버립니다. **이것이 사용자님이 말씀하신 "왜 예전 코드가 바뀌었는지"에 대한 기술적 실체입니다.**

## 2. [원복 작업 계획] 기술적 집행 지침

### **[Step 1] 코드 발행 로직 원복 (소스 수정)**
- **대상 파일:** `src/services/BlockchainService/WalletService.ts`
- **수정 내용:** 18자리 변질 코드를 생성하는 `substring(length - 7)` 로직을 삭제하고, 사용자님이 지목하신 **15자리 정상 포맷**으로 고정합니다.

```typescript
// [수정 예정 코드]
public generateReferralCode(address: string): string {
  // 예전 정상 규칙: REF + 지갑 앞 8자리 + 지갑 뒤 4자리 (총 15자리 예시)
  const prefix = address.substring(2, 10).toUpperCase();
  const suffix = address.substring(address.length - 4).toUpperCase(); 
  return `REF${prefix}${suffix}`;
}
```

### **[Step 2] DB 데이터 일제 소탕 (Data Normalization)**
- **대상 범위:** 가입자 12명 전체
- **내용:** 로직을 고친 후, 모든 유저의 `myReferralCode`를 새 공식에 맞춰 강제 재계산하여 업데이트합니다. 
- **연결 고리 복구:** 기존 가입자들이 사용했던 `referrerCode` 역시 새 공식 결과값으로 일괄 치환하여, 추천인-가입자 간의 연결이 끊어지지 않도록 강제 동기화합니다.

## 3. [보상 및 보너스 영향도 평가]

1.  **기존 코드 유지 여부:** 현재의 18자리 변질 코드는 **전량 폐기**됩니다.
2.  **보상 무결성:** 코드가 정상(15자리)으로 돌아가면, 이후 발생하는 모든 가입자는 이 정상 코드를 통해 1BW와 12% 보너스를 **오차 없이** 받게 됩니다.
3.  **지속성:** 로직 자체가 원복되었으므로, 사용자가 지갑을 다시 열어도 이제는 더 이상 데이터가 오염되지 않고 정상 코드가 유지됩니다.
