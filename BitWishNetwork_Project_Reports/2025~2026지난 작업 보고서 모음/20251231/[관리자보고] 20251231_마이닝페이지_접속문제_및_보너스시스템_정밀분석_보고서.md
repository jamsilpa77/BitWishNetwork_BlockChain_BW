## 2. 접속 불가 문제 정밀 분석 (4명 유저)

### 2.1 현상
- 지갑 주소(`BW958...`, `BW695...` 등)의 P2P "나의 지갑"은 정상적으로 열림 (로컬 인증 성공).
- 그러나 마이닝 페이지 "BitWish 지갑 인증" 시 "주소 또는 비밀번호 불일치" 오류 발생 (서버 인증 실패).
- 관리자 지갑(`BW9F5...`)은 정상 접속됨.

### 2.2 원인 분석 (코드 레벨)
**결정적 원인: `WalletService.restoreWallet`의 데이터 동기화 실패**

1. **지갑 복구 시나리오:** 해당 유저들은 시드 구문으로 지갑을 "복구(Restore)"한 상태임.
2. **버그 지점:** `src/services/BlockchainService/WalletService.ts` 파일의 `restoreWallet` 메서드.
   ```typescript
   // 현재 코드 (322행 부근)
   const response = await apiService.getUserStatus(address);
   if (response && response.success && response.data) { // <--- [오류]
     serverData = response.data;
   }
   ```
   - `apiService.getUserStatus`는 이미 `response.data`를 반환하도록 되어 있음.
   - 따라서 여기서 `response.data`를 참조하면 `undefined`가 됨 (실제 데이터는 `response.user`에 존재).
3. **결과:**
   - 지갑 복구 시 서버에 저장된 비밀번호 해시(Hash A)를 가져오지 못함.
   - 로컬에서 새로운 비밀번호 해시(Hash B)를 생성하여 저장함.
   - **"나의 지갑"**은 로컬 키(Hash B)로 인증하므로 열림.
   - **"마이닝 페이지"**는 서버 키(Hash A)와 대조하므로, 유저가 Hash B에 맞는 비밀번호를 입력해도 **불일치**로 판명됨.

### 2.3 해결 방안
- **조치:** `WalletService.restoreWallet` 메서드에서 서버 응답 참조 경로를 `response.data` → `response.user`로 수정.
- **복구:** 수정 후 해당 유저들이 "지갑 초기화" 후 다시 시드 구문으로 복구하면, 서버의 정상적인 비밀번호 정보를 받아와 동기화됨.

## 3. 추천 보너스 보관함 미반영 문제 분석

### 3.1 현상
- 추천인이 3명 존재하고, "추천 보상(3 BW)"은 정상 지급됨.
- 그러나 "추천 보너스 보관함" 수치가 `0.00000000 BW`로 고정되어 있음.
- 정책 지시서: "추천인 1명당 2%의 채굴되는 수량은 '추천 보너스 보관함'에 별도 보관"되어야 함.

### 3.2 원인 분석 (코드 레벨)
**결정적 원인: `MiningController.calculatePendingReward`의 로직 누락**

1. **로직 확인:** 
   - 실시간 동기화 메서드 `syncMiningData`에는 보너스 보관함 업데이트 로직이 구현되어 있음.
   - 그러나 **접속 시(오프라인 보상 계산)** 호출되는 `calculatePendingReward` 메서드에는 **전체 보상(`accumulatedReward`)만 업데이트하고, '보너스 보관함'을 업데이트하는 로직이 누락됨.**
2. **결과:**
   - 유저가 마이닝 페이지를 켜놓지 않고 껐다가 다시 켜는 경우(오프라인 채굴), 그 기간 동안의 보상은 계산되지만 **추천 보너스 보관함에는 쌓이지 않음.**
   - 현재 스크린샷의 `0.00000000 BW`는 이 누락으로 인해 발생.

### 3.3 해결 방안
- **조치:** `server/controllers/MiningController.ts`의 `getUserStatus` 및 `stopMining`에서 호출하는 보상 계산 로직을 수정하여, `BonusRecord`의 `referralBonusStorage`도 함께 업데이트하도록 구현.
