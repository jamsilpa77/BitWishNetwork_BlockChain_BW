## 🏗️ 1. KYC 본인 인증 및 에어드랍 원칙 확인 (User Directives)

사용자께서 지시하신 **"KYC를 통과해야만 실제 코인으로 지갑에 편입된다"**는 원칙은 현재 시스템의 설계 철학(`Migration Policy`)과 100% 일치함을 확인했습니다.

1.  **가상 포인트 vs 실제 자산:** 모든 채굴 포인트와 추천 보상은 초기 가입 시 '가상 포인트(Pending/Locked)' 상태로 존재하며, 이는 BitWish 블록체인의 실제 자산이 아닙니다.
2.  **KYC 통과 필수:** 가입자가 KYC 본인 인증을 완료하고 관리자가 이를 승인(`APPROVED`)해야만 해당 가입자로 인해 발생한 보상(1BW 및 2% 추가분)이 '사용 가능 금액(Available)'으로 전환됩니다.
3.  **허위 보상 방어:** 개발자가 임의로 코드를 조작하여 1BW를 생성 하더라도, **상대방이 실제 인물로서 KYC를 통과하지 못하면 그 1BW는 영구히 '보관함(Vault)'에 갇혀** 실제 지갑으로 이동할 수 없습니다.

## 🛑 2. 현시점 코드 레벨 무결성 분석 (Code-Level Verification)

`BonusRecord.ts`, `ReferralBonusService.ts`, `types/index.ts` 파일을 분자 단위로 분석한 결과입니다.

### 2-1. 정밀 보관함(Vault) 분리 시스템 확인
*   **보관함 구조:** `referralRewardStorage`(누적 총량)와는 별개로, 실제 사용 가능 여부를 판가름하는 `lockedRewardStorage`(잠긴 보상)와 `availableRewardStorage`(사용 가능 보상) 필드가 물리적으로 분리되어 있습니다.
*   **검증 결과:** 가입 즉시 지급된 1BW는 즉시 `available`로 가는 것이 아니라, **`locked` 상태로 보관함에 먼저 박제**됩니다.

### 2-2. KYC 상태 기반 마이그레이션 로직 (Step-by-Step)
*   **파일:** `ReferralBonusService.ts` 내 `refreshLockStatus` 메서드
*   **작동 원리:** 
    1.  현재 시간과 가입자의 `kycApprovedAt`(KYC 승인 일시)을 비교합니다.
    2.  `kycStatus === 'APPROVED'` 등급이 확인되고 15일의 락업 기간이 경과한 데이터만 필터링합니다.
    3.  조건 충족 시에만 `locked` -> `available`로 금액을 물리적으로 이동(Migration) 시킵니다.
*   **결론:** 관리자가 KYC를 승인하지 않으면 이 로직은 절대 실행되지 않으므로 허위 보상 사용이 원천 불가합니다.

### 2-3. 초 단위 정밀 이력 추적 시스템 (Timestamp Tracking)
*   **파일:** `BonusRecord.ts` (`referralList`), `types/index.ts` (`ReferredUser`)
*   **데이터 필드:**
    *   `joinedAt / joinedDate`: 가입 시점 (년, 월, 일, 시, 분, 초)
    *   `kycApprovedAt`: KYC 승인 시점 (년, 월, 일, 시, 분, 초)
*   **분석 결과:** 모든 보상 내역은 가입자의 지갑 주소와 함께 **초 단위 타임스탬프**를 포함하여 저장되므로, 관리자가 언제 가입했고 언제 승인되었는지 원장에서 완벽하게 추적할 수 있도록 구현되어 있습니다.
