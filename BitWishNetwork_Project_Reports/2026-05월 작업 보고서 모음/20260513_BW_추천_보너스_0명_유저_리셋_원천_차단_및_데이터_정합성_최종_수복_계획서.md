# [2026-05-13] BW 추천 보너스 (0명) 유저 리셋 원천 차단 및 데이터 정합성 최종 수복 계획서

## 1. 결함의 핵심 분석 및 문제점 상세 설명 (Detailed Root Cause Analysis)

### **1.1 현상 상세: "부모는 있고 자식은 없는(0명)" 유저의 무한 리셋**
제공해주신 표본 및 실데이터 전수 조사 결과, 리셋 현상은 오직 특정 조건을 가진 유저에게만 집중적으로 발생하고 있습니다.

1.  **발생 대상**: 본인은 추천인(부모)을 통해 가입하여 **2% 보너스율**을 정상적으로 획득했으나, 아직 본인이 추천한 사람(자식)은 없는 **추천 보너스 (0명)** 상태의 유저.
2.  **비발생 대상**: 자식이 1명이라도 존재하여 명확하게 **추천 보너스 (1명 이상)**로 등재된 유저.
3.  **결함의 실체 (Trigger)**:
    *   **클라이언트**: 부모 혜택(2%)에 따른 보너스를 실시간으로 정상 계산하여 메모리에 유지함.
    *   **서버 (MiningController.ts:151행)**: `if (state.referralCount > 0)` 라는 조건문이 게이트키퍼 역할을 수행 중.
    *   **충돌**: 자식이 0명인 유저는 위 조건문에 걸려 **서버에서 보너스 계산 및 DB(BonusRecord) 저장이 통째로 생략**됨.
    *   **결과**: 마이닝 페이지 동기화 시 서버는 DB의 구버전 데이터(0원)를 반환하고, 클라이언트는 이 0원을 정답으로 믿고 자신의 데이터를 덮어씀으로써 **시각적/물리적 리셋**이 완성됨.

## 2. 초정밀 최종 수복 공정 계획 (Supreme Execution Plan)

### **Step 1: 서버 동기화 게이트웨이 조건 교정**
*   **작업 위치**: `server/controllers/MiningController.ts` (약 151행 `syncMiningData` 내부)
*   **상세 계획**: 
    1.  기존 `if (state.referralCount > 0)` 조건을 **`if (new Decimal(state.referralBonusRate).gt(0))`**로 전면 교체.
    2.  자식의 유무와 상관없이, **보너스율이 존재하는 모든 유저**에 대해 서버가 정밀 계산을 수행하고 DB를 갱신하도록 강제함.

### **Step 2: 서버 복구 엔진(Healer)의 로직 완전 개방**
*   **작업 위치**: `server/controllers/MiningController.ts` (`getUserStatus` 및 `syncMiningData` 내부)
*   **상세 계획**:
    1.  기존 복구 조건인 `miningState.referralCount > 0`을 제거.
    2.  단순 조회 시에도 `BonusRecord`가 누락되었거나 불일치할 경우, 무조건 실제 `User` 컬렉션을 전수 조사하여 데이터를 즉시 치유하도록 수정.

### **Step 3: 클라이언트 레거시 통신(referrals.json) 완전 차단**
*   **작업 위치**: `src/services/BonusService/ReferralBonusService.ts`
*   **상세 계획**:
    1.  이미 폐기된 `/api/storage/load/referrals.json` 호출 로직을 완전 삭제.
    2.  클라이언트가 독자적으로 판단하여 로컬 스토리지로 덮어쓰는 행위를 금지하고, 오직 서버 API의 정규 데이터만 신뢰하도록 데이터 흐름 통제.

## **[초정밀 상세 작업 완료 보고 (Completion Report)]**

### **1. 원천 문제 진단 (Root Cause Diagnosis)**
*   **논리적 사각지대 (Logical Blind Spot)**: 기존 서버 로직은 "추천한 자식이 있는 유저"에게만 보너스 저장 기능을 열어주었습니다. 하지만 BitWish 시스템의 특성상 부모가 있는 유저(0명 유저)는 자식이 없어도 2% 보너스를 받는데, 서버가 이들을 **"보너스 계산 대상이 아님"**으로 분류하여 DB 업데이트를 누락시켰습니다.
*   **클라이언트의 독단적 리셋**: 클라이언트(`ReferralBonusService`)가 이미 존재하지 않는 서버 파일(`referrals.json`)을 호출하려다 실패(404)하자, **비어있는 로컬 스토리지 데이터를 정답인 양 서버 데이터 위에 덮어씌워 버리는** 치명적인 백업 로직의 오작동이 확인되었습니다.

### **2. 단계별 수복 결과 (Step-by-Step Resolution)**
*   **[서버] 관문 교정 (Gatekeeper Fix)**: `syncMiningData`의 진입 조건을 `referralCount > 0`에서 `referralBonusRate > 0`으로 변경하였습니다. 이를 통해 자식이 0명인 유저도 2% 보너스 적립의 혜택을 100% 누릴 수 있도록 서버 통로를 정상화하였습니다.
*   **[서버] 수복 엔진(Healer) 전면 가동**: 유저가 로그인하거나 데이터를 동기화할 때마다 실제 `User` 컬렉션의 추천인 수를 **무조건 재카운트**하도록 설계하였습니다. 설령 DB에 0명으로 잘못 기록되어 있더라도, 접속하는 즉시 실제 데이터로 자동 치유(Auto-Healing)되는 체계를 구축하였습니다.
*   **[클라이언트] 독소 제거 (Legacy Removal)**: 리셋을 유발하던 구식 파일 통신과 로컬 스토리지 강제 덮어쓰기 로직을 **완전 삭제**하였습니다. 이제 화면상의 모든 수치는 서버의 MongoDB 데이터만을 "유일한 진실"로 신뢰하며, 클라이언트의 독단적인 데이터 변경은 불가능하도록 차단되었습니다.

### **3. 최종 정합성 검증 (Verification)**
*   **0명 유저 테스트**: 부모만 존재하는 유저가 마이닝 페이지를 수차례 닫고 열어도, 서버에서 수복된 2% 보너스와 보관함 수치가 변함없이 유지됨을 확인하였습니다.
*   **불일치 해소**: 서버 시간 기준의 정밀 연산(Decimal.js 50자리)이 모든 동기화 과정에 적용되어, 클라이언트와 서버 간의 데이터 오차를 원천적으로 차단하였습니다.
