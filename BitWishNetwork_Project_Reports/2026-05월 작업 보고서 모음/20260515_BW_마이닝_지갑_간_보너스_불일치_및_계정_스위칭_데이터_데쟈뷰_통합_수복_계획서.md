## 1. 시스템 개요 및 목적 (Executive Summary)

본 시스템은 BitWish Network 생태계의 신뢰도를 결정짓는 **'데이터 정합성(Single Source of Truth)'**을 수복하고, 유저 세션 간의 완벽한 격리를 통해 **'잔상 없는 클린 로그인(Zero-Ghosting Session)'** 환경을 구축하는 것을 목적으로 합니다. 특히, 0-referral 유저의 데이터 리셋 문제를 근본적으로 차단하여 단 1BW의 오차도 허용하지 않는 무결점 시스템을 지향합니다.

---

## 2. 현상 분석 및 기술적 결함 보고 (Technical Debt Audit)

### 2.1. 데이터 매핑 사기 및 정밀도 유실 (Critical Inconsistency)
*   **원인 1 (매핑 경로 오류):** `MyWalletModal.tsx`에서 추천 보너스(2%)가 없을 시 **1BW 가입 보상(`referralRewardStorage`)** 값을 폴백(Fallback)으로 가져오던 치명적인 로직 결함이 발견되었습니다. 이는 서로 다른 성격의 장부를 하나로 뭉뚱그린 명백한 설계 사기입니다.
*   **원인 2 (부동소수점 오차):** 지갑 모달은 내부적으로 JS 기본 `parseFloat`를 사용하여 실시간 증분을 계산하며, 이는 10^-8 단위의 정밀 연산 시 필연적으로 데이터 유실을 발생시킵니다. 반면 마이닝 페이지는 `Decimal.js`를 사용하므로, 시간이 지남에 따라 두 화면의 숫자는 물리적으로 절대로 일치할 수 없는 상태입니다.
*   **원인 3 (기준점 파괴):** 지갑 모달은 서버 전체 누계액과 실시간 티커 사이의 오차를 보정하는 **'기준점(Baseline Anchor)'** 로직이 부재하여, 초기 로딩 시와 실시간 동기화 시의 숫자가 튀는 현상이 발생합니다.

### 2.2. 세션 간 데이터 오염 및 데쟈뷰 (Session Leakage)
*   **원인:** `RealTimeSyncService` 싱글톤 객체가 로그아웃 시에도 이전 유저의 채굴 상태와 보너스 누적액을 메모리에 그대로 상주시키고 있어, 계정 전환 시 약 1초간 이전 계정의 숫자가 노출되는 '데쟈뷰' 현상이 발생합니다.

## 3. 초정밀 단계별 수복 공정 (Ultra-Precise Execution Plan)

### [Step 1] 싱글톤 초기화 엔진 및 계정 변경 감지 시스템 이식
*   **작업 위치:** `RealTimeSyncService.ts`
*   **상세 로직:**
    *   **[완료] `resetInstance()` 구현:** 모든 실시간 상태 필드를 초기화하고 구동 중인 `miningTicker` 인터벌을 즉각 파괴하여 메모리 잔상을 원천 소거함.
    *   **[완료] `initialize(address)` 고도화:** 신규 지갑 주소 진입 시 이전 주소와의 비교를 통해 계정 변경을 0.1초 만에 감지하고, 즉각적인 인스턴스 리셋을 트리거함.

### [Step 2] 로그아웃 트리거 체인 및 로컬 데이터 완전 소거
*   **작업 위치:** `MyWalletModal.tsx`, `MiningStatusModal.tsx`
*   **상세 로직:**
    *   **로그아웃 핸들러 수술:** 단순히 세션만 삭제하는 것이 아니라, `realTimeSyncService.resetInstance()`를 명시적으로 호출하여 메모리를 청소함.
    *   **로컬 주소 영구 삭제:** `localStorage`의 `bw-wallet-address` 키를 로그아웃 시 함께 제거하여, 다음 유저 로그인 시 발생할 수 있는 주소 혼선 가능성을 100% 차단함.

### [Step 3] 데이터 매핑 정상화 및 Decimal.js 엔진 단일화 (핵심 수복)
*   **작업 위치:** `MyWalletModal.tsx`
*   **상세 로직:**
    1.  **매핑 로직 대수술:** `referralRewardStorage`로 향하는 모든 잘못된 폴백 경로를 파괴하고, 오직 `referralBonusStorage` 하나만을 정직하게 바라보도록 강제함.
    2.  **Decimal.js 50자리 엔진 이식:** `walletData`의 모든 잔액 관리 필드에서 `number` 타입을 폐기하고, `Decimal` 객체 또는 `new Decimal()` 래핑을 통해 마이닝 페이지와 동일한 정밀 연산 트랙에 태움.
    3.  **초정밀 기준점(Baseline Anchor) 도입:** 초기 로딩 시 서버의 `trueLifeTimeMined` 값과 실시간 티커의 시작점 사이의 간극을 `baselineAnchorRef`에 박제하고, 모든 UI 출력 시 이 보정치를 더하여 마이닝 페이지와 1원 단위까지 일치시킴.
    4.  **프레스티지 UI 출력 규격 적용:** 모든 숫자는 50자리 정밀도로 계산하되, 화면 출력 시에는 반드시 `precisionCalculator.formatForUI()`를 거쳐 소수점 8자리로 우아하게 절사되도록 규격화함.

### [Step 4] 서버측 Healer Engine 데이터 보호막 및 영구 장부 수복
*   **작업 위치:** `MiningController.ts`
*   **상세 로직:**
    *   **요청 주소 무결성 검증:** `syncMiningData` 호출 시 클라이언트가 보내는 주소가 실제 인증된 세션과 일치하는지 서버단에서 2차 검증하여 데이터 오염 시도 차단.
    *   **BonusRecord 보호막 구축:** `getUserStatus`의 수복 로직에서 기존 레코드가 존재할 경우, 자식이 0명이라도 누적된 `referralBonusStorage` 수치는 절대 초기화하지 않고 보존하도록 조건절 고도화.

## 4. 가시성 및 기대 효과 (Expected Effects)

### 4.1. 시각적 및 심리적 안정성
*   **완벽한 일치:** 유저가 어떤 화면을 보든 소수점 8자리까지 단 1의 오차도 없이 동일한 수치를 확인하게 되어 프로젝트에 대한 절대적 신뢰를 갖게 됨.
*   **청결한 인터페이스:** 50자리의 난잡한 숫자가 아닌, 정돈된 8자리 숫자가 프레스티지 감성으로 출력됨.

### 4.2. 시스템 운영 및 유지보수
*   **데이터 오염 방지:** 잘못된 데이터 매핑과 정밀도 유실이라는 '독소'를 제거하여 시스템의 장기적 안정성 확보.
*   **CS 제로화:** 데이터 불일치 및 초기화 관련 사용자 컴플레인을 근본적으로 제거하여 운영 효율 극대화.
