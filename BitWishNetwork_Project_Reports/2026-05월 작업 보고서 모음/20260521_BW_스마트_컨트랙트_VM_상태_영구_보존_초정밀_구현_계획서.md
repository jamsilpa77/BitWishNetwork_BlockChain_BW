# BitWishNetwork 스마트 컨트랙트(VM) 상태 영구 보존 초정밀 구현 계획서

**작성일**: 2026년 05월 21일
**작업 목표**: 트랜잭션 종료 시 메모리 상에서 휘발되어 버리는 스마트 컨트랙트 가상머신(VM)의 Storage 결함을 초정밀 수술하여, 트랜잭션 원자성(Atomicity)을 보장하는 완벽한 영구 보존(State Persistence) 아키텍처 이식.

---

## 1. 🔍 [상세 분석] 무결점 이식 전략 (Storage Persistence Data Flow 설계)
현재 블록체인의 코어인 `BitWishBlockchain.ts`의 `executeTransaction` 메서드 내부(401~421 라인 부근)에는 가상머신(VM)을 호출하는 로직이 구현되어 있으나, 정작 컨트랙트의 실행 결과(예: 장부 잔액 변동, 소유권 이전 등)를 저장할 `contractStorage` 객체가 1회성 메모리 `Map`으로 선언된 후 휘발되고 있습니다.
이는 블록체인 스마트 컨트랙트의 가장 기본이 되는 "상태 보존(State Persistence)" 원칙을 완전히 위배하는 치명적인 결함입니다. 
따라서 본 계획서는 **"트랜잭션 실행 전 DB에서 상태 트리(State Trie)를 로드하고, VM 실행이 100% 성공(Success)했을 때만 DB에 덮어쓰기(Commit)를 수행하여 데이터 오염을 원천 차단하는 완벽한 스토리지 아키텍처"**를 이식하는 데 목적이 있습니다.

### 📍 2공정: `executeTransaction` 심장부 Storage 로드 및 커밋(Commit) 수술
현재 빈 깡통으로 선언된 `contractStorage` 로직을 완전히 걷어내고, 앞서 만든 DB 연동 및 해시(storageRoot) 갱신 로직을 주입합니다.
*   **작업 파일**: `BitWishNetwork_BlockChain/src/engine/BitWishBlockchain.ts` (`executeTransaction` 내부)
*   **상세 로직**:
    ```typescript
    // [기존 코드 - 삭제]
    // const contractStorage = new Map<string, string>(); // TODO: 실제 DB Storage 로드
    // const vm = new BitWishVM(contractStorage, transaction.gasLimit);
    // const result = vm.execute(targetAccount.codeHash + " " + (transaction.data || ""));

    // [신규 코드 - 초정밀 이식]
    // 1. 상태 로드 (State Load)
    const contractStorage = await this.loadContractStorage(transaction.to);
    const vm = new BitWishVM(contractStorage, transaction.gasLimit);
    
    // 2. 가상머신(VM) 실행
    const result = vm.execute(targetAccount.codeHash + " " + (transaction.data || ""));

    if (!result.success) {
        // 에러 발생 시 DB 덮어쓰기 로직을 타지 않음 (완벽한 Rollback/방어)
        return { success: false, error: 'VM 실행 실패 (가스 초과 또는 로직 오류)' };
    }

    // 3. 상태 커밋 및 storageRoot 갱신 (State Commit)
    const newStateObj = Object.fromEntries(contractStorage);
    const newStorageRoot = crypto.createHash('sha256').update(JSON.stringify(newStateObj)).digest('hex');
    
    targetAccount.storageRoot = newStorageRoot;
    await this.saveContractStorage(transaction.to, contractStorage); // DB 영구 보존
    
    return { success: true };
    ```
    *주의*: `executeTransaction` 내부 로직이 비동기(`await`)로 변경됨에 따라 메서드 시그니처를 `async`로 변환하고, 이를 호출하는 블록 마이닝 로직(`BitWishMiner.ts` 등)의 동기화 파이프라인도 맞물려 수술합니다.

### 📍 3공정: 가상머신(VM) 코어 해시 엔진(SHA3) 활성화
VM 내부에서 데이터를 암호화하는 `SHA3` 명령어(Opcode)가 단순 문자열 합치기 등의 목업(Mock)으로 방치된 것을 실제 Node.js 암호화 모듈로 교체하여 블록체인의 무결성을 확보합니다.
*   **작업 파일**: `BitWishNetwork_BlockChain/src/engine/BitWishVM.ts`
*   **상세 로직**:
    ```typescript
    import * as crypto from 'crypto';

    // [기존 목업 로직 삭제 및 신규 해시 엔진 이식]
    // VM execute 루프 내 op === 'SHA3' 처리 부분:
    const data = this.stack.pop();
    if (data) {
        const hash = crypto.createHash('sha256').update(data).digest('hex');
        this.stack.push(hash);
    }
    ```

### 📍 4공정: 전역 무결성 및 사이드 이펙트 최종 검증
수술 완료 후 컨트랙트 배포 및 호출 트랜잭션을 발생시켜 다음을 엄격히 검증합니다.
1. **원자성(Atomicity) 검증**: 트랜잭션 에러(가스 초과, 로직 오류 등) 발생 시 `result.success`가 `false`로 떨어지며 `saveContractStorage` 로직이 완벽하게 차단되어 기존 DB가 오염되지 않는가?
2. **해시 무결성 검증**: 트랜잭션 성공 시 `accounts` 컬렉션의 `storageRoot` 해시값이 저장된 데이터와 1:1로 일치하게 변경되는가?
3. **상태 영속성 검증**: 트랜잭션이 성공한 후, 다음 트랜잭션에서 동일 컨트랙트를 재호출할 때 업데이트된 `contractStorage`를 정확하게 불러와 이전 상태 위에서 로직이 이어지는가?
