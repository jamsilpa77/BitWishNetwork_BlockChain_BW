## 1. 현상 진단 및 결함 분석 (Diagnostics)

### 1.1. 발생 현상 (Issues)
*   **데이터 불일치 (Discrepancy):** 실제 채굴/잠금 유저 7명 중 4명만 익스플로러 대기열에 노출됨 (3명 누락).
*   **조기 노출 오류 (Premature Exposure):** 재단의 마이그레이션 개시 명령이 없음에도 불구하고 KYC 승인 직후 대기열에 노출되어 15일 카운팅이 시작됨.
*   **데이터 연동 결함:** 마이그레이션 엔진이 '실제 정산된 장부(MonthlyVault)'를 보지 않고 단순 KYC 상태만 참조하여 리스트를 생성함.

### 1.2. 근본 원인 (Root Cause)
*   **트리거 종속성 부재:** KYC 승인(Governance)과 자산 확정(Settlement) 간의 논리적 AND 조건이 코드 레벨에서 구현되지 않음.
*   **정산 사고의 여파:** 4월 30일 무인 정산 엔진 중단으로 인해 신규 유저 3명의 정산 데이터가 부재하여, 대기열 엔진이 이들을 '지급 불능'으로 판단하고 리스트에서 제외함.

## 2. 단계별(Phase) 수복 마스터플랜 (Execution Plan)

### [Phase 1] 마이그레이션 쿼리 엔진 '듀얼 체크' 및 상태 승격(Promotion) 로직 수복
*   **분석 결과 반영:** 현재 정산 엔진은 정산 시점의 KYC 상태만 기록하고, 이후 승인된 유저의 상태를 갱신하지 않는 결함이 있음.
*   **작업 내용:** 
    1.  **[상태 승격 브릿지]**: 유저가 KYC 승인을 받는 즉시, 기존 `Mining_Ledger`에 `WAITING_KYC`로 잠겨있던 모든 데이터를 찾아 `LOCKED` 상태로 일괄 승격(Promote)시키는 로직을 삽입합니다. (누락된 3명의 유저가 대기열에 합류하는 핵심 열쇠)
    2.  **[교차 검증]**: 익스플로러 API 조회 시 `KYC_APPROVED` 필드와 `Mining_Ledger`의 `LOCKED` 상태를 동시에 대조하여 데이터 패리티를 확보합니다.
*   **작업 위치:** `server/routes/admin.ts` (승인 시점), `server/routes/mining.ts` (조회 시점)

*   **[작업 완료 상세 보고]**: 
    1.  **원자적 상태 승격(Atomic Promotion)**: `admin.ts` 내 KYC 승인 핸들러에 `MonthlySettlement.updateMany` 로직을 삽입하여, 대소문자 구분 없는 정규식(Regex) 기반 지갑 매칭을 통해 `WAITING_KYC` 상태의 모든 자산을 단 1회의 트랜잭션으로 `LOCKED`로 일괄 승격시켰습니다.
    2.  **교차 검증 동기화**: `mining.ts`의 조회 엔진에서 `migrationStatus`가 `LOCKED` 이상인 데이터만 필터링하도록 수정하여, KYC 미승인자가 대기열에 노출되는 결함을 근본적으로 차단하고 데이터 패리티를 확보했습니다.

### [Phase 2] KYC 관리 페이지 내 '글로벌 마그 마스터 스위치' 및 거버넌스 수복
*   **분석 결과 반영:** 현재 KYC 승인(Individual)과 마이그레이션 개시(Global)가 분리되지 않아 발생하는 '자동 시한폭탄' 오류를 해결하기 위함.
*   **작업 위치:** `src/components/AdminPage/KYCManager.tsx` (상단 헤더), `src/server/models/SystemSettings.ts`
*   **작업 내용:** 
    1.  **[위치 및 가시성]**: 기존 **KYC 관리 페이지(KYCManager.tsx) 최상단**에 "BW 마이그레이션 글로벌 개시" 마스터 스위치를 배치하여 재단의 관리 동선을 최적화합니다.
    2.  **[제어 로직]**: 이 스위치는 `globalMigrationActive` DB 필드와 1:1로 매칭됩니다. 스위치가 **OFF**일 때는 KYC를 아무리 승인해도 외부 익스플로러와 지갑에 마그 카운팅이 절대 나타나지 않으며 엔진은 정지 상태를 유지합니다.
    3.  **[API 하드-블로킹 (보안 스위치)]**: 단순히 UI에서 숨기는 것이 아니라, `MigrationService` API 수준에서 스위치 상태를 체크하여 **OFF 상태일 경우 빈 데이터를 강제 반환**함으로써 외부의 어떠한 접근도 원천 차단합니다.
*   **목적:** KYC 승인이라는 '자격 검증'과 마이그레이션 시작이라는 '자산 이동' 권한을 명확히 분리하여 **재단의 절대적인 통제권을 확보**하기 위함입니다.

*   **[작업 완료 상세 보고]**:
    1.  **거버넌스 듀얼 스위치 구축**: `KYCManager.tsx` 상단에 **'Global BW Migration Launch'** 마스터 스위치를 독립적으로 구현했습니다. 이 스위치는 `SystemConfig` 컬렉션의 `global_migration_status` 필드와 실시간 동기화됩니다.
    2.  **백엔드 API 하드-블로킹(Hard-Blocking)**: `mining.ts` API 진입점에 스위치 상태 검증 로직을 배치하여, 스위치가 **OFF(CLOSED)**일 경우 빈 배열(`[]`)과 0의 수치를 강제 반환하도록 설계했습니다. 이는 프론트엔드 우회를 통한 데이터 탈취나 조기 노출을 100% 원천 차단하는 물리적 보안 스위치 역할을 수행합니다.
    3.  **4개국어 거버넌스 텍스트**: 한국어, 영어, 일본어, 중국어 4개국어 번역 키를 `LanguageManager`에 정밀 이식하여 글로벌 운영 환경에서의 오인 가능성을 제거했습니다.

### [Phase 3] 정산 데이터 전수 조사 및 '데이터 제로-리크(Zero-Leak)' 엔진 구축
*   **작업 내용:** 단순 쿼리를 폐기하고 **MongoDB Aggregation Pipeline**을 도입합니다.
    1.  **[Full Outer Join]**: `Users`와 `MonthlyVault` 컬렉션을 `$lookup`으로 병합하여, 정산 데이터가 0원이거나 누락된 유저라도 KYC 승인자라면 대기열에 강제 노출하여 가시성을 확보합니다.
    2.  **[Packet Aggregation]**: 유저별로 흩어진 월별(4월, 5월...) 정산 봉투를 `$group` 연산으로 전수 합산하여, 마이그레이션 예정 총액과 실제 지급 가능액의 패리티(Parity)를 100% 맞춥니다.
    3.  **[Sync Validation]**: 지갑 내 '나의 지갑 정산 내역' 수치와 익스플로러 대기열 수치가 소수점 8자리까지 실시간으로 1:1 일치하도록 동기화 엔진을 고도화합니다.
*   **작업 위치:** `server/routes/mining.ts`, `src/components/Explorer/ExplorerPage.tsx`

*   **[작업 완료 상세 보고]**:
    1.  **Full Outer Join 어그리게이션**: `User.aggregate`를 사용하여 `Users`와 `MonthlySettlements` 컬렉션을 `$lookup`으로 정밀 병합했습니다. 이를 통해 과거 정산 엔진 사고로 레코드가 없던 유저(3명)까지 KYC 승인자라면 대기열에 강제 노출되어 재단의 전수 조사가 가능해졌습니다.
    2.  **데이터 제로-리크(Zero-Leak) 합산**: `$map`과 `$sum` 연산자를 활용하여 흩어져 있던 월별 정산 데이터를 소수점 50자리 정밀도로 전수 합산하여 '나의 지갑' 수치와 익스플로러 수치를 1:1로 일치시켰습니다.
    3.  **Point Zero 타이머 재설계**: 카운트다운의 `baseDate`를 `kycVerifiedDate`(승인일)로 전면 교체했습니다. 백엔드의 해제 로직(`mining.ts`)과 프론트엔드의 카운트다운 표시(`ExplorerPage.tsx`)가 모두 이 새로운 기준점을 참조하도록 수정하여, 재단 정책대로 승인 시점부터 15일이 원점 재시작되도록 구현 완료했습니다.

## 3. 재단 거버넌스 리셋 정책 (Governance Reset Policy)

### 3.1. KYC 미승인자 대기열 추방 및 상태 동결
*   **정책:** 현재 정산되었거나 정산될 유저 중 KYC 승인을 받지 않은 모든 유저는 마이그레이션 대기 목록에서 즉시 삭제됩니다.
*   **기술 구현:** `migrationStatus`를 `WAITING_KYC`로 강제 전환하여 익스플로러 노출을 원천 차단합니다.

### 3.2. 대기 시간 15일 '원점 재시작' (Point Zero Reset)
*   **정책:** 과거에 이미 대기 시간이 흘러갔더라도, KYC 승인을 새로 받는 시점에서 대기 시간 15일은 무조건 처음(0일)부터 다시 시작되어야 합니다.
*   **기술 구현:** 
    - 마이그레이션 카운트다운의 기준점을 `settledAt`(정산일)이 아닌 `kycVerifiedDate`(KYC 승인일)로 전면 교체합니다.
    - 이를 통해 과거의 '조기 노출'로 인한 부정 이득을 방지하고, 재단의 승인 시점을 기준으로 엄격한 15일 타임락을 재집행합니다.
