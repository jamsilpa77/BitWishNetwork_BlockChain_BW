**프로젝트명**: BitWish Community (Standalone Multi-Window Project)

## ⚠️ 1. 절대 준수 및 금지 규칙 (Core Security Principles)

본 커뮤니티 프로젝트는 다음의 절대 규칙 하에 한 치의 오차 없이 통제된 환경에서 개발됩니다.

*   **🛡️ 마이닝 생태계 철통 방어:** 마이닝 데이터베이스(MongoDB 하이브리드) 및 기존 코드는 0.1%도 침범하거나 간섭하지 않습니다.
*   **🪟 완벽한 멀티윈도우 격리:** 커뮤니티는 홈페이지 내 전용 멀티윈도우 팝업(Drag & Drop 가능) 내부에서만 구동되며, 외부(부모) 컴포넌트와 상태를 일절 공유하지 않습니다.
*   **❌ 전역 오염 원천 차단:** 전역 상태 라이브러리, 전역 모달, 공통 변수, 공통 함수 사용을 절대 금지합니다.
*   **🌍 다국어 및 UI 독립:** 커뮤니티 윈도우 내부에 자체적인 **4개국 다국어 번역(i18n)** 시스템과 **다크/나이트 모드**를 완벽하게 분리하여 구현합니다.

## 🚀 2. 10대 개발 공정 마스터 로드맵 (The 10-Step Plan)

최신형 하이브리드 커뮤니티(DC+레딧+블라인드 스타일)를 완성하기 위한 1~10번 공정의 정밀 계획입니다.

### [Phase 1: 기반 아키텍처 및 DB 설계]
*   **1공정: Prisma DB 전체 Schema 설계**
    *   마이닝 DB와 완전히 물리적으로 분리된 커뮤니티 전용 독립 DB 스키마 설계.
    *   `User`(로그인/회원가입), `Post`(카테고리형 게시글), `Comment`(대댓글), `Reaction`(이모지 반응) 구조화.
*   **2공정: JWT 및 소셜 로그인(구글) 구현**
    *   지갑 연동이 아닌, 일반적이고 완벽한 커뮤니티 전용 JWT 로그인/회원가입 로직 개발.
    *   보안이 강화된 Access/Refresh 토큰 아키텍처 설계.

    **✅ [Phase 1 초정밀 공정 완료 보고서 (2026-05-18)]**
    *   **작업 내용:** 
        1. `BW_Community_Server`라는 100% 물리적 격리 폴더 신설 완료.
        2. `schema.prisma`를 통해 User, Post, Comment, Reaction 간의 완벽한 RDBMS 무결성 관계 세팅 완료.
        3. `authController.ts`, `authRoutes.ts`, `jwtHelper.ts`를 신설하여 15분(Access)/7일(Refresh) 듀얼 토큰 및 구글 OAuth 라우팅 구조 캡슐화 완료.
    *   **가시성(Visibility):** 
        기존 마이닝 폴더 코드에 단 1byte의 간섭 없이 독립된 커뮤니티 전용 백엔드 뼈대와 인증 API 라우터(`/api/auth/...`)가 즉각 통신 가능한 상태로 투명하게 가시화되었습니다.
    *   **기대 효과(Expectation) 및 효율(Efficiency):** 
        마이닝 서버와의 메모리/상태 간섭이 원천 차단(Air-Gap)되었습니다. 향후 Phase 2~4 개발 중 커뮤니티 서버에 치명적 버그나 트래픽 폭주가 발생해도, 마이닝 채굴 티커는 0.1초의 지연 없이 100% 정상 구동됩니다. 코드베이스 분리로 서버 컴파일 및 재시작 속도가 극대화되며, 전역 오류 확산 확률을 0%로 만들었습니다.


### [Phase 2: API 코어 및 실시간 엔진]
*   **3공정: 전체 Node.js 백엔드 실제 코드 구축**
    *   RESTful 설계 원칙에 입각한 게시글 CRUD 및 댓글 로직 컨트롤러/라우터 완성.
*   **4공정: Socket.io 실시간 인기글 시스템 (Hot Score)**
    *   `좋아요*3 + 하트*5 + 웃겨요*2 - 싫어요*2` 알고리즘 탑재.
    *   실시간 반응 감지 및 랭킹 자동 갱신 웹소켓 엔진 장착.
*   **5공정: 모바일 앱 연동 API (Headless Architecture)**
    *   모든 백엔드 응답을 순수 JSON API로 출력하여, 추후 웹/모바일 앱 어디서든 100% 호환되도록 대통합 설계.

    **✅ [Phase 2 초정밀 공정 완료 보고서 (2026-05-18)]**
    *   **작업 내용:** 
        1. `postController.ts`, `commentController.ts`를 신설하여 상태가 없는(Stateless) 순수 JSON Headless API(게시글/댓글 CRUD) 뼈대 구축 완료.
        2. `socketManager.ts`를 신설하여 전역 상태 오염을 완벽히 차단한 캡슐화된 웹소켓(Socket.io) 인스턴스 환경 구성 완료.
        3. `reactionController.ts` 내부에 `hotScore` 계산 공식과 웹소켓 Broadcasting 이벤트를 완벽히 통합 이식 완료.
    *   **가시성(Visibility):** 
        이제 커뮤니티 서버는 100% 물리적으로 독립된 상태에서 모바일/웹 공용 API를 뿜어낼 수 있으며, 사용자가 리액션을 남길 때마다 `hot-post-update` 이벤트가 즉각적으로 방출되어 실시간으로 인기글 랭킹을 뒤바꿀 준비가 완료되었습니다.
    *   **기대 효과(Expectation) 및 효율(Efficiency):** 
        가장 강력하고 복잡한 백엔드 코어(실시간 엔진 및 API)가 마이닝 로직과 단 1%의 충돌도 없이 완벽하게 구축되었습니다. 데이터베이스 연산(Prisma)과 실시간 통신(Socket)이 독립 공간에서만 처리되므로, 향후 수만 명의 유저가 커뮤니티에서 폭주해도 BitWish 메인넷은 티끌만큼의 서버 부하도 받지 않는 압도적인 안정성과 효율을 달성했습니다.


### [Phase 3: 독립형 프론트엔드 구축]

    *   기존 기본값인 `3000`포트 대신, 실제 마이닝 프론트엔드 서버가 점유하고 있는 `5000`포트를 커뮤니티 서버(`index.ts`)의 CORS 허용 도메인(`http://localhost:5000`)에 정밀 주입하여 브라우저의 통신 거부(ERR_CONNECTION_REFUSED) 및 보안 차단 현상을 원천 종식.

    **✅ [Phase 4.5 초정밀 공정 완료 보고서 (2026-05-19)]**
    *   **작업 내용:** 
        1. Webpack이 Tailwind CSS 유틸리티 클래스를 컴파일하지 못하는 근본 문제를 타파하고자, `BWCommunityWindow.tsx` 파일 내부에 캡슐화된 동적 `<style>` 태그 스타일시트를 직접 주입하여 UI를 복구했습니다.
        2. `:hover`, `:focus` 등 바닐라 CSS의 강력한 가상 클래스와 스피너 회전 애니메이션(`@keyframes spin`)까지 스타일시트에 담아 마이닝 본체에 빌드 간섭 없이 구동되도록 구현했습니다.
        3. 커뮤니티 독립 백엔드 서버(`BW_Community_Server/src/index.ts`)의 CORS 허용 도메인에 실제 마이닝 프론트엔드가 활성화되어 작동 중인 `http://localhost:5000` 주소를 완벽하게 이식했습니다.
    *   **가시성(Visibility):** 
        이제 커뮤니티 버튼을 클릭하면 메인 홈페이지의 레이아웃 흐름을 훼손하지 않는 완벽한 독립 플로팅 윈도우 창이 absolute/fixed 위치에 렌더링됩니다. 또한, 다크/라이트 테마 변경에 따라 배경색과 글자색, 보더의 경계선 색상이 브라우저 내에서 실시간 반응형으로 변경됩니다.
    *   **기대 효과(Expectation) 및 효율(Efficiency):** 
        마이닝 프론트엔드 프로젝트에 Tailwind CSS 관련 패키지를 설치하거나 Webpack 설정을 개조하지 않아 빌드 안전도가 100% 보존됩니다. 프론트엔드 포트 불일치(3000 vs 5000)로 발생하던 브라우저 CORS 보안 거부 현상이 완전히 소멸되어, 프론트엔드와 독립 백엔드(4000) 포트 간의 REST API 및 데이터 폴링이 완전 무결하게 개통되었습니다.

### [Phase 4.5-2: 커뮤니티 백엔드 API & DB의 5001포트 완전 통합 수복 공정]
*   **원인 규명 및 반성:**
    *   마이닝 시스템은 5000포트(프론트)와 5001포트(백엔드)로 유기적 기동되는 단순·안전한 아키텍처임에도 불구하고, 제가 엉뚱하게 포트 4000 독립 서버(`BW_Community_Server`)를 만들어 개발 위치를 격리시키는 큰 오판을 저질렀습니다. 이로 인해 서버가 꺼져 있어 `Failed to fetch` 통신 거부 장애를 냈고, 카테고리 및 UI 규격이 설계 사양서와 맞지 않고 예외 가이드가 누락되는 총체적 불일치를 초래했습니다.
*   **15공정: 커뮤니티 전용 Mongoose 스키마 5001포트 백엔드 내재화**
    *   기존 마이닝 시스템의 데이터베이스(MongoDB) 구조를 전혀 건드리지 않고 완벽하게 보호하기 위해, `BitWishNetwork_MiningSystem/server/index.ts` 내부에 커뮤니티 전용 컬렉션 모델(`CommunityUser`, `CommunityPost`, `CommunityComment`, `CommunityReaction`)을 Mongoose 스키마로 새롭게 설계·삽입합니다.
    *   기존 회원(`User`) 컬렉션과 완전히 분리된 샌드박스형 회원 체계를 활용하여 마이닝 메인넷 연산 자원 및 기존 로직에 미치는 영향을 0.00%로 영구 억제합니다.
*   **16공정: 5001포트 Express 내 커뮤니티 API 개통**
    *   5001포트의 Express 메인 인스턴스에 `/api/community` 라우터를 추가 마운트하여, 로그인/가입, 게시글 목록/상세, 댓글/대댓글, 리액션 투표(좋아요, 하트, 웃겨요, 싫어요), 관리자 통제(공지사항 고정, 악성 유저 차단, 게시글 블라인드 삭제) API를 완전 구현합니다.
*   **17공정: 프론트엔드 포트 연동 수복 및 디자인/UX 마감 고도화**
    *   `communityClient.ts` 내 통신 대상을 `http://localhost:5001/api/community`로 변경하여 별도 서버 기동 없이 5001포트 메인 백엔드를 타고 실시간 폴링 및 데이터 교환이 에러 없이 흐르도록 수복합니다.
    *   카테고리 탭을 설계 사양서에 명시된 한국어/다국어 규격인 **유머, 정보, 자유, 질문, 게임, 익명, 공지**로 수복하고, 비로그인 시 경고 얼럿 발생 및 로그인 창 이동 동선 가드를 장착합니다.
    *   헤더 바를 [BW 익스플로러] 및 [랭킹 게시판]의 고급 다크/라이트 테마 헤더 규격(`ranking-header-bar`)과 1:1 동일한 가로폭 마진, 대칭 정렬 레이아웃으로 변경하여 시각적 고급감을 완성합니다.
