# [관리자 분석요청 세밀 분석 보고서] 20251230

## 1. 개요 및 분석 범위
본 보고서는 관리자님의 엄중한 지시에 따라 **BitWishNetwork_MiningSystem**의 현재 코드 상태를 **원자 단위**까지 쪼개어 분석한 결과입니다. 
분석 기준은 **`BitWsihNetwork BW채굴 마이닝 홈페이지 및 정책 개발 지시서.md`**의 모든 조항이며, 절대 필자의 주관이나 임의적 판단을 배제하고 **코드에 적힌 사실(Fact)**만을 기반으로 작성되었습니다.

## 2. 정밀 분석 결과 (Code Level Deep Dive)

### A. **서버 사이드 (`server.js`) - 데이터 저장소 엔진의 치명적 결함**
1.  **지시사항 위반 (개별 독립 저장소 미준수):**
    -   지시서 Line 67: *"유저는 각각 개개인의 마이닝 페이지는 완벽한 독립성 보장 및 각 완전한 데이터베이스... 가지게 하고"*
    -   **현실태:** `server.js` Line 534 `const allUsers = this.getAllUsers();`
    -   **분석:** `getAllUsers()`는 디렉토리 내 모든 파일을 읽어 메모리 배열로 만드는 함수입니다. 유저 가입 시마다 **모든 유저 파일을 강제로 읽어들이는** 미친 구조로 되어 있습니다. 이는 **"천만 명 확장"** 지시를 정면으로 위반하는 구조이며, 유저가 늘어날수록 서버는 100% 멈추게 됩니다.
    -   **판정:** **구조적 불일치 (CRITICAL FAIL)**

2.  **피라미드 종속성 구현의 한계 (Line 574 ~ 597):**
    -   **현실태:** `while` 루프를 돌며 `referrerId`를 추적해 `totalBonusRate`를 더하는 로직 자체는 존재합니다.
    -   **문제점:** Line 601 `this.saveUserData(ancestor)`를 반복문 밖에서 일괄 처리하려고 배열(`updatedAncestors`)에 담습니다. 만약 상위 100번째 추천인까지 간다면 메모리에 100개의 객체를 띄워놓고 한 번에 쓰기를 시도하는데, 이 과정에서 **서버 다운 시 데이터 증발(Data Loss)** 위험이 매우 큽니다.
    -   **판정:** **로직 불안정 (UNSTABLE)**

3.  **보관함 분리 미흡 (Line 360 ~ 362):**
    -   **현실태:** `rewardStorage`와 `bonusStorage` 필드는 존재하지만, `join` 로직(Line 555)에서 하드코딩된 값(`1.0`)만 더하고 있을 뿐, 실제 채굴 시 발생하는 **2%의 보너스가 `bonusStorage`에 실시간으로 쌓이는 연산 로직(Tick Logic)**이 `api/mining/tick` 엔드포인트에 **전무(None)**합니다.
    -   **판정:** **기능 미구현 (MISSING FEATURE)**

### B. **클라이언트 사이드 (`MiningStatusModal.tsx`) - 서버 절대 복종 위반**
1.  **로컬 우선의 반역 코드 (Line 197 ~ 211):**
    -   **현실태:**
        ```typescript
        const localState = localStorage.getItem('bw-mining-state');
        if (parsed.isMining === false) trustLocal = true;
        // ...
        if (trustLocal) { setIsMining(false); ... console.log('...서버 데이터보다 우선시합니다.'); }
        ```
    -   **분석:** 관리자 지시서 Line 51 *"모든 보너스 시스템은 마이닝 시작 후에만 작동"* 및 서버 데이터 신뢰 원칙을 무시하고, 로컬 스토리지에 '중지'라고 적혀 있으면 서버가 '채굴 중'이라고 해도 **강제로 화면을 꺼버리는** 코드가 적발되었습니다.
    -   **판정:** **지시 불이행 (MUTINY)**

2.  **호버 데이터 가짜 표기 (Line 544):**
    -   **현실태:** `<span className="value-hover" data-full={userStats.baseRate.toFixed(8)}>`
    -   **분석:** `userStats.baseRate`는 이미 소수점 처리가 된 상태(`Decimal` 객체)지만, 근본적으로 서버에서 받아올 때 `toFixed(4)` 또는 문자열로 잘려서 오는 경우가 있어, 진정한 50자리 정밀도가 UI까지 살아서 오지 못하고 있습니다.
    -   **판정:** **정밀도 손실 (LOSS OF PRECISION)**

### C. **서비스 계층 (`ReferralBonusService.ts`) - 이중 장부 문제**
1.  **현실태:** `localStorage`에 `bw-referral-data`라는 키로 데이터를 별도 저장하고 관리하며, 이를 `MiningStatusModal`과 동기화하려고 애쓰는 코드가 가득합니다.
2.  **분석:** 서버(`server.js`)가 이미 파일 DB(`users/*.json`)를 가지고 있는데, 프론트엔드가 별도의 장부(`localStorage`)를 쓰는 것은 **"데이터 불일치"**의 근원입니다. 지시서의 "서버 절대 복종"은 프론트엔드가 데이터를 **"가지는(Own)"** 게 아니라 **"보여주는(Display)"** 역할만 해야 함을 의미합니다.
3.  **판정:** **설계 오류 (ARCHITECTURAL FLAW)**

## 3. [공식] 통합 수정 작업 계획 보고서 (6단계)

관리자님의 명령에 따라, 위에서 적발된 문제점들을 6단계로 나누어, 한 단계가 완벽히 검증된 후 다음 단계로 넘어가는 **단계별 정밀 타격 수정 계획**을 수립합니다.

### **[Step 1] 독립 저장소 엔진(BISE) 탑재 및 메모리 의존성 제거**
-   **목표:** `server.js`에서 `this.referrals` 배열 및 `getAllUsers()` 호출 **전면 삭제**.
-   **작업:**
    -   `loadUserData(wallet)`: 오직 해당 지갑 파일 하나만 읽는 함수로 최적화.
    -   `/api/referral/join`: 전체 유저 스캔 로직을 폐기하고, 입력된 추천인 코드의 주인(Owner) 파일만 **인덱싱(Indexing) 없이 직접 조회**하는 로직으로 변경.
    -   **검증:** 1번 지갑의 파일만 수정해도 2번 지갑 파일에 전혀 영향이 없는지 파일 시스템 레벨 확인.

### **[Step 2] 보관함 물리적/논리적 완전 분리**
-   **목표:** `rewardStorage`(1BW)와 `bonusStorage`(2% 채굴분)의 연산 로직 분리.
-   **작업:**
    -   `server.js`의 유저 데이터 스키마 확정:
        -   `rewardStorage`: 가입 발생 시 `+1.0` (이벤트성)
        -   `bonusStorage`: 채굴 Tick 발생 시 `+ (기본채굴량 * 0.02 * 추천인수)` (실시간 누적성)
    -   `/api/mining/tick`에 보너스 계산 로직(Line 404 부근) **신규 삽입**.
    -   **검증:** 채굴을 1분 돌렸을 때 `bonusStorage` 수치만 올라가고 `rewardStorage`는 고정되어 있는지 확인.

### **[Step 3] 피라미드 무한 종속 엔진(Recursive Propagation) 구현**
-   **목표:** A -> B -> C 연결 시 C가 가입하면 B와 A의 파일이 모두 열려서 수정되는 로직 완성.
-   **작업:**
    -   `server.js` 내 `updateAncestors(userId, depth)` 재귀 함수 구현.
    -   **Atomic Write:** 부모 파일 열기 -> `totalBonusRate` +2 수정 -> 즉시 닫기 (메모리에 쌓아두지 않음).
    -   관리자 지갑(최상위)까지 끊김 없이 올라가는지 테스트.
    -   **검증:** C 가입 직후 A의 `totalBonusRate`가 즉시 +2 되었는지 파일 조회.

### **[Step 4] 프론트엔드 반역 코드 숙청 및 서버 복종화**
-   **목표:** `MiningStatusModal.tsx`의 로컬 스토리지 우선 로직 삭제.
-   **작업:**
    -   `trustLocal`, `localStorage` 의존 코드 삭제.
    -   Line 70~230의 초기화 로직을 **"API 호출 -> `setUserStats`"** 단방향 흐름으로 통일.
    -   **검증:** 로컬 스토리지 다 지우고 새로고침 해도 서버 데이터 그대로 뜨는지 확인.

### **[Step 5] 30번 지시서(접속제한 해제) 및 인증 정상화**
-   **목표:** "내 폰 아니어도 내 지갑+비번이면 접속" 구현.
-   **작업:**
    -   `MiningAuthModal.tsx`의 주소 비교 `if (storedAddress !== inputAddress)` 삭제.
    -   서버 `/api/user/status`를 호출하여 `secondPasswordHash` 일치 여부만 확인.
    -   **검증:** 시크릿 모드(새 기기 환경)에서 기존 지갑 주소+비번으로 로그인 성공 여부 확인.

### **[Step 6] 50자리 초정밀 UI 및 호버링 완성**
-   **목표:** 겉은 4자리, 속은 50자리 데이터 흐름 완성.
-   **작업:**
    -   서버에서 `res.json` 보낼 때 `toFixed(4)` 하지 말고 원본 문자열 전송.
    -   클라이언트 `Decimal.js`로 받아서, 화면 표시 때만 `format(4)` 처리, 호버 툴팁(`title` or custom tooltip)에는 원본 문자열 표시.
    -   **검증:** 마우스 올렸을 때 0.00000000... 정밀 값 노출 확인.
