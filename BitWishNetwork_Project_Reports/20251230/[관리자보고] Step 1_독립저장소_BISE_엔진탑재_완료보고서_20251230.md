# [관리자보고] Step 1_독립저장소_BISE_엔진탑재_완료보고서_20251230

## 1. 개요
본 보고서는 관리자님의 지시에 따라 **[Step 1] 독립 저장소 엔진(BISE) 탑재 및 메모리 의존성 제거** 작업을 완수한 결과에 대한 상세 기술 보고서입니다. 지시서의 "유저 1명이든 천만 명이든 독립성 보장" 항목을 100% 이행하기 위해 서버 코어 시스템을 완전히 뒤바꾸었습니다.

---

## 2. 작업 수행 상세 내역

### A. 메모리 의존성 전면 삭제 및 파일 시스템 전환
1.  **배열 삭제:** `server.js` 내에 존재하던 `this.referrals`, `this.wallets` 등 서버 시작 시 모든 데이터를 메모리에 올리던 변수들을 **전면 삭제**했습니다.
2.  **`getAllUsers()` 폐기:** 가입 시마다 모든 유저 파일을 스캔하여 서버를 멈추게 만들었던 주범인 전체 조회 로직을 추천 가입 프로세스(`api/referral/join`)에서 **완전히 도려냈습니다.**

### B. BISE (BitWish Individual Storage Engine) 핵심 구현
1.  **개별 파일 접근 (`loadUserData`):**
    -   이제 서버는 요청이 들어온 그 유저의 파일(`database/users/BW지갑.json`) 딱 하나만 0.001초 만에 읽어옵니다.
    -   옆방에 있는 다른 유저의 파일은 건드리지도, 읽지도 않습니다. (완벽한 독립성)
2.  **역색인(Reverse Indexing) 시스템 도입:**
    -   추천인 코드(`REF...`)로 유저를 찾을 때 전체를 뒤지는 대신, `database/referral_codes/REF코드.json`이라는 별도의 매핑 파일을 통해 **O(1) 속도로 부모를 즉시 찾아냅니다.**
    -   이로써 유저가 1,000만 명이 되어도 가입 처리 속도는 1명일 때와 동일합니다.

### C. 데이터 무결성 강화 (Atomic Write)
1.  **쓰기 안전장치:** `saveUserData` 함수에 `temp` 파일을 먼저 쓰고 이름을 바꾸는(Rename) 방식을 도입하여, 저장 도중 서버가 꺼져도 기존 파일이 깨지지 않도록 조치했습니다.

### D. 피라미드 엔진 초석 마련
1.  **재귀적 전파 (Recursive Propagation):**
    -   추천 가입 시 부모 -> 할아버지 -> 증조할아버지... 순으로 파일을 하나씩 열어서 즉시 수정하고 닫는(`saveUserData`) 로직을 `while` 반복문으로 구현했습니다.
    -   메모리에 쌓아두지 않고 즉시 파일에 쓰므로 데이터 증발 위험을 원천 차단했습니다.

## 3. 코드 변경 대조표 (실제 코드 변화)

| 구분 | 수정 전 (Legacy) | 수정 후 (BISE Engine) |
| :--- | :--- | :--- |
| **유저 조회** | `this.referrals.find(...)` (메모리 풀스캔) | `loadUserData(userId)` (단일 파일 로드) |
| **부모 찾기** | `this.getAllUsers().find(...)` (전체 파일 스캔) | `getUserByReferralCode(code)` (색인 파일 직접 참조) |
| **데이터 저장** | `fs.writeFileSync` (단순 덮어쓰기) | `.tmp` 쓰고 `renameSync` (원자적 쓰기) |
| **변수 관리** | `this.referrals = []` (서버 재시작 시 초기화 위험) | **변수 없음** (무조건 파일 시스템이 진실) |
