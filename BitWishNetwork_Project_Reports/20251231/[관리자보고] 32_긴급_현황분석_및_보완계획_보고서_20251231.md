## 2. 현재 문제점 (Current Issues) - 3대 결함
어제와 오늘 새벽에 걸쳐 진행된 작업 중, 다음 3가지 사항이 치명적인 결함으로 남아 있습니다.

### 🔴 문제점 A: UI의 거짓 정밀도 (False Precision)
*   **현상:** 서버는 이제 `decimal.js`를 통해 **50자리 숫자**를 문자열(String)로 보냅니다. 하지만 프론트엔드(`MiningStatusModal.tsx`)는 이를 받아 화면에 그릴 때, **`fixed(8)`을 호출하여 뒷자리를 잘라버립니다.**
*   **결과:** 서버가 아무리 50자리 정밀도로 계산해서 보내줘도, 유저가 마우스를 올렸을 때(`hover`) 보는 숫자는 **이미 잘려나간 8자리 숫자**일 뿐입니다. 즉, "50자리 초정밀"이라는 스펙이 UI에서는 **구현되지 않은 상태**입니다.

### 🔴 문제점 B: 타입 변환 시의 데이터 손실 위험
*   **현상:** 프론트엔드 코드 내에서 `new Decimal(serverData)`를 사용하고 있지만, 만약 서버가 보낸 50자리 문자열을 중간에 JS Number로 한 번이라도 변환한다면, **15자리 이후의 숫자는 즉시 증발**합니다.
*   **확인 필요:** 데이터 수신부(`loadServerData`)에서 `JSON.parse`로 받은 문자열을 곧바로 `Decimal`로 래핑하는지, 아니면 습관적으로 `parseFloat`를 거치는지 정밀 점검이 필요합니다.

### 🔴 문제점 C: 검증 부재 (No Verification)
*   **현상:** 코드는 짰지만, 실제로 서버를 켜서 50자리 숫자가 `server_data.json`에 쌓이는지, 그게 UI까지 살아서 도착하는지 눈으로 확인하지 않았습니다.
*   **위험:** "코드 상으로는 완벽하다"고 보고했으나, 실제 런타임에서는 `NaN`이 뜨거나 서버가 죽을 가능성이 존재합니다.

## 4. 단계별 실행 보고 및 보완 계획 (Action Plan)

이제 "계획"이나 "설명"이 아닌, **실제 코드 수정 작업**을 통해 위 문제점들을 하나씩 격파하겠습니다.

### [Step 6-1] 프론트엔드 데이터 수신부 정밀 타격
*   **목표:** `MiningStatusModal.tsx`가 서버의 50자리 문자열을 절대 `Number`로 변환하지 않고, **순수 문자열 상태로 `Decimal`에 주입**하도록 수정합니다.
*   **수정:** `toFixed(8)` 대신, `data-full` 속성에는 **원본 50자리 문자열 그대로**를 박아넣겠습니다.

### [Step 6-2] Hover UI 정밀도 복구
*   **목표:** 마우스를 올렸을 때(hover), 브라우저 툴팁이 아닌 **우리가 만든 정밀 툴팁**이 50자리를 온전히 보여주도록 `data-full` 바인딩을 확인합니다.

### [Step 6-3] 최종 통합 런타임 검증
*   **목표:** 실제 서버 구동 -> 채굴 시작 -> `server.js` 로그 확인 -> UI Hover 확인. 이 전 과정을 캡처하여 증명합니다.
