## 1. 현재 시스템 vs 관리자님 요구사항 (Gap Analysis)

관리자님의 7가지 요구사항을 기준으로, 현재 코드(`c:\BitWishNetwork_BlockChainMainnet`)의 상태를 냉정하게 평가했습니다.

### 1-1. 지갑 생성과 즉시 연결 (Gap 존재)
*   **관리자님 요구:** "나의 지갑이 생성되면, 그 즉시 마이닝 인증 모달이 지갑 주소를 영구적으로 인지해야 한다."
*   **현재 코드 상태:** 
    *   지갑 생성(`createWallet`) 시 `apiService.registerUser`를 호출합니다.
    *   **문제점:** 이 호출이 실패(네트워크 오류, 서버 에러 등)했을 때, **재시도(Retry)하거나, 다음 로그인 시 자동으로 복구하는 로직이 완성도 있게 구현되지 않았습니다.**
    *   이로 인해 지금의 5개 지갑처럼 "지갑은 있는데 마이닝 페이지는 모르는" 사태가 발생합니다.
*   **필요한 조치:** `MyWalletModal`이 로드될 때, 로컬 스토리지에 지갑이 있다면 **자동으로 서버에 존재 여부를 묻고, 없다면 즉시 무소음(Slient)으로 등록시키는 '자동 동기화' 로직**이 추가되어야 합니다.

### 1-2. 보안 (2차 비밀번호 필수) (구현됨, 보완 필요)
*   **관리자님 요구:** "인증은 하되, 2차 비밀번호 없이는 절대 마이닝 페이지를 열 수 없어야 한다."
*   **현재 코드 상태:**
    *   `MiningAuthModal`에서 비밀번호 입력창을 띄우고 있습니다.
    *   **문제점:** DB에 데이터가 없는 경우, 비밀번호를 검증할 대상(해시값)조차 없어서 "인증 실패"가 뜹니다. 이는 사용자에게 혼란을 줍니다.
*   **필요한 조치:** DB에 데이터가 없더라도, 로컬 스토리지의 지갑 정보(해시)를 이용하여 **우선 로컬 검증을 통과하면, 그 정보를 바탕으로 서버 DB를 즉시 생성(복구)하고 문을 열어주는 유연함**이 필요합니다.

### 1-3. 데이터 영구 저장 (MongoDB) (구현됨, 안정화 필요)
*   **관리자님 요구:** "시드문구, 비밀번호, 추천코드, 지갑주소가 MongoDB에 영구히 저장되어야 한다."
*   **현재 코드 상태:**
    *   MongoDB 스키마(`User`, `MiningState`, `BonusRecord`)는 잘 잡혀 있습니다.
    *   **문제점:** 데이터 저장(`save`) 시점에 중복 에러가 발생하면 저장이 취소되는 등의 **예외 처리**가 미흡했습니다. (1월 7일 수정 완료됨)
*   **필요한 조치:** 데이터 저장이 **'트랜잭션(Transaction)'** 수준으로 안전하게 처리되도록, 가입 로직이 실패하면 절대로 넘어가지 않게 **강력한 검증**을 덧붙여야 합니다.

## 2. 보안/수정/구현해야 할 핵심 포인트 (Action Plan)

"엉뚱한 파일 생성" 없이, 기존 파일들의 **연결 고리(Link)**를 강화하는 작업입니다.

### 2-1. [수정] `WalletService.ts`: "자동 입주 신고 시스템"
*   **위치:** `restoreWallet` 및 `registerUser` 호출부.
*   **내용:** 
    1.  사용자가 "지갑 인증 모달"을 띄웠을 때, 로컬에 지갑이 있는가? -> (Yes)
    2.  서버 DB에 이 지갑이 있는가? (`getUserStatus`) -> (No)
    3.  **그렇다면 묻지도 따지지도 말고 즉시 `registerUser`를 실행하여 DB에 등록해라.** (비밀번호 입력 전에 미리 등록해두는 것이 아니라, 비밀번호 입력 후 검증 성공 시 등록이 없으면 생성하는 방식)
    *   **핵심:** 사용자는 별도의 "복구" 절차를 밟을 필요가 없습니다. 그냥 평소처럼 비밀번호 치고 로그인하면, 시스템이 알아서 "어? DB에 없네? 바로 만들어 줄게" 하고 처리해야 합니다.

### 2-2. [수정] `UserController.ts`: "유연한 가입 수용"
*   **위치:** `register` 함수.
*   **내용:** 이미 지갑 주소가 DB에 있어도 에러를 뱉지 말고 **"업데이트(덮어쓰기)"** 하거나 **"무시(성공 처리)"**하여, 프론트엔드가 당황하지 않게 해야 합니다. (1월 7일 기수정 완료, 재확인 필요)

### 2-3. [구현] `MiningAuthModal.tsx`: "One-Stop 인증"
*   **위치:** 프론트엔드 로그인 모달.
*   **내용:** 
    *   로그인 버튼 클릭 시 -> 1. 로컬 비밀번호 검증(우선) -> 2. 서버 로그인 시도 -> 3. (서버에 없다면) -> 4. **자동 가입(Register) 후 로그인 성공 처리.**
    *   이 로직이 들어가면 현재 5명의 유저는 **그냥 비번 치고 로그인 버튼만 누르면 해결**됩니다.

## 3. 최종 결론

관리자님의 말씀 ("지갑 만들면 즉시 영구 등록되어야 한다")은 시스템 설계의 **'제1원칙'**입니다.
현재 시스템은 이 원칙을 지키기 위한 뼈대는 있지만, **연결 부위의 나사가 풀려 있어서(자동 등록 로직 부재)** 5명의 유저가 "미아"가 된 것입니다.

**해결책은 명확합니다.**
별도의 복구 페이지를 만들거나 이상한 짓을 할 필요가 없습니다.
**"로그인 시도 시, DB에 없으면 즉시 자동 등록하고 문을 열어준다."**
이 로직 하나만 `MiningAuthModal`과 `WalletService`에 심으면, 모든 것이 관리자님의 구상대로 **완벽하게 연결**됩니다.
