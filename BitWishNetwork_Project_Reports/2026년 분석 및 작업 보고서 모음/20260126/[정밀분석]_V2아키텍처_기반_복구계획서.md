# [최종상세] BitWish 시스템 정상화 및 V2 아키텍처 기반 정밀 구현 시방서

**작성일시:** 2026년 1월 26일
**작성자:** BitWish 시스템 담당자
**문서 성격:** 추상적인 계획을 배제하고, **파일 단위의 수정 내역, 로직의 흐름, 예상되는 결과**를 명확히 기술한 **실행 지침서**.

---

## 1. 개요: 무엇을, 왜 고치는가?

### **1.1. 현황 진단**
*   **문제:** 현재 백엔드(`server.ts`)는 V2 설계도에 명시된 `routes`, `controllers`, `models` 구조 없이 단일 파일에 모든 로직이 뒤섞여 있습니다. 이로 인해 **추천 보너스 계산, 실시간 반영, 관리자 조회** 기능이 구현될 공간이 없어 작동하지 않고 있습니다.
*   **목표:** 설계도(V2)대로 폴더 구조를 물리적으로 생성하고, 그 안에 **핵심 비즈니스 로직(보너스, 실시간, 관리자)**을 코드로 구현하여 채워 넣습니다.

---

## 2. 상세 구현 계획 (Step-by-Step)

### **2.1. [Step 1] 백엔드 물리적 구조 복원 (Architecture Restoration)**

**[작업 내용]**
*   `deploy_backend/src/server/` 경로 하위에 다음 3개 디렉토리를 생성합니다.
    1.  **`routes/`**: 클라이언트의 요청(URL)을 해당 로직으로 연결해주는 **배선함** 역할.
    2.  **`controllers/`**: 실제 데이터 계산, 저장, 조회 등 일처리를 하는 **작업장** 역할.
    3.  **`models/`**: MongoDB에 데이터가 저장될 **규격(Schema)**을 정의하는 역할.

**[기대 효과]**
*   코드가 역할별로 분리되어, 관리자 기능이나 보너스 로직을 추가할 때 서로 엉키지 않고 명확하게 구현할 수 있게 됩니다.

---

### **2.2. [Step 2] 데이터베이스 스키마(Model) 정의**

**[작업 내용: `server/models/` 파일 생성]**
1.  **`MiningState.ts`**:
    *   **필드:** `walletAddress` (PK), `accumulatedReward` (총 채굴량), `referralCount` (추천인 수).
    *   **목적:** 유저의 **실시간 채굴 상태**를 영구 저장. 단순히 파일에 쓰는 게 아니라 DB에 구조화된 데이터로 저장함.
2.  **`BonusRecord.ts`**:
    *   **필드:** `referrer` (추천인 주소), `referee` (가입자 주소), `bonusType` (2% or 1BW), `timestamp`.
    *   **목적:** **누가 누구를 추천했는지(족보)**를 기록하여, 보너스 계산 및 관리자 추적 시 사용.

**[기대 효과]**
*   단순 파일(`storage.ts`)이 아닌 **관계형 데이터**로 관리되므로, "내 하위 추천인이 몇 명인지", "언제 가입했는지"를 SQL처럼 정확하게 조회할 수 있습니다.

---

### **2.3. [Step 3] 핵심 로직 구현 (Controller Implementation)**

**[작업 내용: `server/controllers/MiningController.ts` 작성]**

**① 추천 보너스 계산 로직 (Dynamic Calculation)**
*   **로직:** 유저가 `/sync` 요청을 보낼 때마다 다음을 수행합니다.
    1.  `BonusRecord` 테이블에서 **내 지갑 주소가 추천인(referrer)으로 등록된 건수**를 `Count` 합니다.
    2.  **공식 적용:** `NewRate = Base(0.25) + (Count × 0.02)`.
    3.  만약 `Partner` 상태라면 `1.25`를 추가로 더합니다.
*   **결과:** 추천인이 10명 늘어나면, 다음번 `sync` 때 즉시 채굴 속도가 `0.25 -> 0.45`로 껑충 뜁니다. (영구적이고 즉각적임)

**② 실시간 전역 통계 반영 (Socket Broadcast)**
*   **로직:** `sync` 처리가 끝난 직후, 증가된 채굴량(`delta`)을 서버 메모리 변수(**`GlobalTotalIssued`**)에 더합니다.
*   **전파:** `io.emit('mining_update', { total: GlobalTotalIssued })`를 호출하여 모든 접속된 클라이언트(홈페이지)에 뿌립니다.
*   **결과:** 유저가 채굴기를 돌리는 순간, 홈페이지 대시보드의 숫자가 **실시간으로 파라락** 올라갑니다.

---

### **2.4. [Step 4] 관리자 페이지 기능 연결 (Admin API)**

**[작업 내용: `server/routes/admin.ts` 및 `AdminController.ts` 작성]**

**① 지갑 주소 정밀 조회**
*   **기능:** 관리자가 지갑 주소를 검색하면 다음 데이터를 한 번에 묶어서 보냅니다.
    *   **기본 정보:** 가입일, KYC 상태 (`User` 모델 조회)
    *   **채굴 현황:** 현재 잔액, 채굴 중 여부 (`MiningState` 모델 조회)
    *   **추천 내역:** **나를 추천한 사람** 및 **내가 추천한 사람 목록** 전체 (`BonusRecord` 모델 조회)
*   **결과:** 관리자 페이지 UI에서 "데이터 없음" 오류 대신, 실제 유저의 **모든 활동 내역이 상세하게 표출**됩니다.

---

### **2.5. [Step 5] 원격 접속 및 사용자 경험 개선 (Frontend Fix)**

**[작업 내용: `WalletService.ts` 빗장 제거]**
1.  **로직 수정:** `login` 및 `setSecondPassword` 함수 내에 있는 **`if (!localWallet) return FAIL`** 코드를 **삭제**합니다.
2.  **대체 로직:**
    *   입력된 **시드 구문**을 해시화하여 서버로 보냅니다.
    *   서버 DB에 있는 해시와 일치하면 **"Pass"** 응답을 줍니다.
    *   Pass 응답을 받으면 프론트엔드는 즉시 **로컬 저장소(localStorage/File)를 재생성**하고 로그인을 승인합니다.
*   **결과:** PC방이나 새 핸드폰에서도 시드 구만만 있으면 **즉시 로그인하고 2차 비번을 설정**하여 채굴을 시작할 수 있습니다.

---

## 3. 최종 산출물 (Definition of Done)

이 작업이 모두 완료되면 시스템은 다음과 같이 변합니다.

1.  **백엔드:** `routes`, `controllers`, `models` 폴더가 생성되고 코드가 꽉 채워져 있습니다.
2.  **마이닝:** 추천인이 늘어나면 채굴 속도가 **2%씩 정확히 증가**하여 17.63 BW 등의 수치가 나옵니다.
3.  **홈페이지:** 숫자가 멈춰있지 않고 **유저 활동에 맞춰 실시간으로 상승**합니다.
4.  **관리자:** 지갑 주소 검색 시 **모든 이력(족보 포함)**이 조회됩니다.
5.  **접속:** 어디서든 **시드 구문으로 로그인**이 가능해집니다.

이 시방서에 따라 **오차 없이 정밀하게** 구현 작업을 진행하겠습니다.
