# 20260419 BitWish Network 프로젝트 잔여 핵심 과제 정리 및 계속 실행 보고서

## 1. 수수료 100% 네트워크 생태계 기금 전환 및 6:4 자산 배분 시스템 (백엔드 코어)

### [문제점]
사용자가 지불하는 거래 및 블록 수수료가 단순히 누군가의 통장에 쌓이거나 의미 없이 사라진다면, 210억 개 한정 수량인 빗위시 생태계 자산의 선순환 구조를 해치고 유저에게는 '박탈감'을 줄 우려가 있습니다.

### [개선 및 구현 방안]
*   **시스템 가로채기 및 자산 전환:** 유저가 코인을 전송할 때 발생하는 수수료를 시스템이 즉시 가로채어, 유통망에서 분리하여 **'전략적 생태계 자산'**으로 전환합니다. 이는 사라지는 것이 아니라 생태계 매니저 지원 및 커뮤니티 육성을 위한 예비 자산으로 보관됩니다.
*   **60:40 내부 디지털 금고(기금) 누적 저장:** 확보된 수수료 자산은 시스템 내부의 특수한 저장 변수(금고) 2곳에 6:4 비율로 자동 배분되어 적립됩니다.
    *   '생태계 가치 보호 기금'에 수수료의 60% 누적 저장
    *   '재단 운영 및 커뮤니티 지원 기금'에 수수료의 40% 누적 저장
*   **작업 파일 위치:** `src/engine/BitWishBlockchain.ts` 안에 있는 핵심 송금 함수(`executeTransaction`) 내의 수수료 분배 및 기금 적립 로직 구현.
### [1. 수수료 100% 네트워크 생태계 기금 전환 및 6:4 자산 배분 시스템 (백엔드 코어) 작업 완료 상세 보고서]

#### A. 개요 및 설계 철학 (Philosophy)
본 작업은 빗위시(BW)의 210억 개 한정 자본을 보호하기 위해, 수수료를 '삭제(소각)'하는 것이 아니라 **'미래 지향적 생태계 자산으로 전환(Transition)'**하는 백엔드 코어 엔진을 구축한 것입니다. 모든 용어와 로직에서 'Burn'을 제거하고 'Fund Accumulation'을 적용하여 네트워크의 신뢰도를 극대화했습니다.

#### B. 기술적 구현 상세 (Technical Details)
1.  **초정밀 데이터 구조 (Precision Data Structure)**
    *   **Ecosystem Vault:** `private ecosystemFund: Decimal` - 생태계 가치 보호를 위한 메인 금고.
    *   **Foundation Vault:** `private foundationFund: Decimal` - 재단 및 커뮤니티 매니저 지원용 금고.
    *   **Total Counter:** `private totalAccumulatedFees: Decimal` - 누적된 총 기금 추적.
    *   *비고: 모든 계산에는 50자리 부동소수점 오차를 차단하기 위해 `Decimal.js` 엔진을 사용함.*

2.  **수수료 가로채기 및 분배 엔진 (Interception & Split Engine)**
    *   **함수:** `executeTransaction(transaction: BitWishTransaction)`
    *   **메커니즘:** 송금 트랜잭션 실행 시 발신자로부터 수수료를 즉시 시스템이 가로채기(Intercept) 함.
    *   **6:4 분배 로직:** 
        *   `ecoShare = fee * 0.6` (생태계 기금 적립)
        *   `foundShare = fee - ecoShare` (재단 기금 적립 - 소수점 잔여 오차 방지를 위해 차감 방식 채택)
    *   **결과:** 수수료는 유통 공급량에서 분리되어 금고로 이동하며, 이는 물리적 파괴가 아닌 **'전략적 보호 예치'** 상태로 전환됨.

3.  **영구 데이터 보전 레이어 (Persistence Layer)**
    *   **저장소:** MongoDB `network_stats` 컬렉션
    *   **보안:** 매 트랜잭션마다 `global_fund_stats` 문서를 업데이트하여 서버 다운 시에도 기금 데이터의 무결성을 보장함.
    *   **복구 로직:** 블록체인 초기화 시 `loadNetworkStats()`를 통해 기존 기금 잔액을 메모리에 완벽히 복원함.

4.  **투명성 제공 인터페이스 (Transparency API)**
    *   `getStats()` 및 `getStatus()` 메소드를 업데이트하여 외부(익스플로러 등)에서 `ecosystemFund`, `foundationFund` 가치를 실시간으로 모니터링할 수 있도록 연동 완료.

#### C. 기대 효과 (Impact)
*   **유저 신뢰:** "내 수수료가 생태계를 위해 재투자된다"는 성장의 확신 부여.
*   **경제 모델:** 한정된 210억 개의 자산이 네트워크 내에서 끊임없이 순환하며 가치를 제고하는 'Circular Economy' 완성.

## 2. 수수료 상위 금고의 '관리자(Administrator) 초절대 권한' 보안 및 원격 직할 결재망

### [문제점]
*   다중 서명(Multi-sig) 방식의 의사결정 지연 및 빗위시 1인 통치 철학과의 불일치 제거 필요.
*   관리자 1인 체제의 유일한 취약점인 '비밀번호 유출/해킹'에 대한 물리적 방어책 부재.
*   침해 사고 발생 시 자금 동결 및 복구에 대한 초정밀 기술 시나리오 부족.

### [정밀 분석: 일반 유저 시스템 vs 슈퍼 어드민 시스템 분리]
*   **논리적 공간(Namespace)의 격리:** 기존 유저들이 사용하는 지갑 생성, 니모닉 발급, 랜덤 문구 입력 모달 로직은 프론트엔드 유틸리티 영역에 상주하며, 제가 구현할 슈퍼 어드민 보안 로직은 백엔드 엔진의 특권 영역(`System Privilege`)에 존재합니다. 두 시스템은 호출 경로 자체가 다르므로 상호 간섭이 발생할 수 없습니다.
*   **기존 마이닝 시스템 보호:** 메인 홈페이지의 마이닝 시작 버튼 및 지갑 주소 대조 프로세스는 현재의 안정적인 코드를 그대로 유지합니다. 슈퍼 어드민 전용 보안키(YubiKey) 인증은 오직 '기금 출금'이라는 특수한 관리자 전용 트랜잭션 시에만 트리거되도록 설계하여 일반 유저의 사용 경험을 완벽히 보호합니다.
*   **기술적 보증:** `WalletService.ts` 등 기존의 유저 지갑 모듈에는 단 한 줄의 코드 수정도 가하지 않으며, 별도의 관리자 보안 API(`admin-secure-api.ts`)를 신설하여 독립적으로 운영함으로써 시스템 무결성을 보장합니다.

### [전문가적 전략 제언: 관리자 실무 준비 사항]
최상급 보안 및 1인 절대 권한의 안정적 운영을 위해 관리자님께 다음의 준비 과정을 강력히 제언합니다.
*   **물리 보안 키(YubiKey 5) 2개 확보:** 시스템 안정성을 위해 메인용과 비상 백업용으로 총 2개의 전용 하드웨어 키를 준비하십시오. 이는 물리적 '진짜 열쇠'가 됩니다.
*   **24단어 마스터키의 오프라인 봉인:** 시스템 구축 시 발급되는 24개의 복구 단어는 절대로 디지털 기기(사진, 카톡 등)에 남기지 말고, **실제 종이에 펜으로 적어 비밀 금고에 보관**하십시오. (USB 분실 시 유일한 자금 복구 수단)
*   **물리적 터치 기반 집행 습관:** 해커는 전 세계 어디서든 관리자님의 PC를 들여다볼 수 있지만, 관리자님 책상 위의 **USB 키 단자를 직접 손가락으로 누르는 행위**는 절대 흉내 낼 수 없습니다. 모든 기금 집행은 반드시 이 '물리적 터치'가 수반되도록 습관화하십시오.

### [개선 및 구현 방안 - 슈퍼 어드민 보안 모델]
1.  **마스터 니모닉(Mnemonic) 생성 및 오프라인 보존:**
    *   블록체인 제네시스 레벨에서 관리자 전용 **'24단어 마스터 복구 코드'**를 생성.
    *   이를 디지털 기기가 아닌 물리적 종이에 기록하여 금고에 보관(Air-gapped)함으로써, 온라인상에서의 키 탈취 가능성을 100% 원천 차단.
2.  **FIDO2 하드웨어 보안 키(YubiKey) 물리 연동:**
    *   오직 관리자님이 소유한 **실물 보안 USB 키**의 '고유 식별값'을 블록체인 엔진에 하드코딩.
    *   기금 인출 시, 관리자님이 USB 키를 직접 터치하여 생성되는 **'물리적 디지털 서명'**이 없으면 시스템은 그 어떤 명령도 거부함. (PC가 해킹당해도 물리적 터치 없이는 인출 불가)
3.  **원스톱 직할 집행 대시보드 (Executive Push):**
    *   관리자 페이지 내에 기금 전송 전용 고속 라인 구축.
    *   관리자님의 **지문/얼굴 인식 또는 보안키 터치** 한 번으로 생태계 기금을 즉각적으로 파트너사나 매니저에게 발송하는 '무결점 초고속 결재 프로세스' 구현.
4.  **독점적 서명 검증 로직 (Administrator Supreme Logic):**
    *   `BitWishBlockchain.ts` 내에 타인이 절대 조작할 수 없는 `verifyAdminSignature` 함수를 삽입하여, 모든 기금 이동의 '최종 승인권'을 관리자 주소에 고정.

*   **작업 파일 위치:** 
    *   `src/engine/BitWishBlockchain.ts`: 관리자 전용 절대 승인 및 하드웨어 서명 검토 로직 주입.
    *   `src/server/admin-secure-api.ts`: 관리자 보안키 연동을 위한 전용 통신 API 신설.
### [2. 수수료 상위 금고의 '관리자(Administrator) 초절대 권한' 보안 및 원격 직할 결재망 작업 상세 완료 보고서]

#### A. 기술적 집도 성과 (Technical Mastery)
1.  **엔진 레벨 '초특권(Supreme)' 필드 각인:**
    *   `BitWishBlockchain.ts`의 심장부에 관리자 마스터 주소와 하드웨어 보안키 ID를 저장할 특수 필드 구축 완료. 
    *   어떠한 외부 조작으로도 이 권한이 변조되지 않도록 MongoDB에 암호화 보존 로직 적용.
2.  **물리적 서명 기반 출금 엔진 (`adminSupremeWithdraw`):**
    *   관리자님의 **실물 보안 키(YubiKey) 터치**로 생성되는 암호학적 서명이 유효할 때만 기금이 인출되는 초정밀 검증 로직 구현.
3.  **격리된 보안 채널 구축 (`admin-secure-api.ts`):**
    *   일반 유저의 웹 트래픽과 완전히 분리된 관리자 전용 보안 통신로를 개설했습니다. 이 통로는 오직 관리자 대시보드와 하드웨어 보안 키 사이의 '신뢰 기반 대화'만 허용합니다.

#### B. 전문가적 전략 제언: 관리자 실전 가이드라인 (중요)
관리자님, 본 보안 시스템의 '최종 열쇠'는 소프트웨어가 아닌 관리자님의 실물 손안에 있습니다. 아래 지침을 반드시 수행하여 보안을 완성하십시오.

*   **행동 1. 하드웨어 키 실물 확보:** 지금 즉시 **'유비키 5(YubiKey 5) 시리즈' 2개**를 준비하십시오. 
    *   *내용:* 하나는 평소 업무용 가방에, 하나는 비상용으로 자택 금고에 보관하십시오. 이것이 빗위시의 모든 부(Wealth)를 지키는 유일한 물리적 문고리입니다.
*   **행동 2. 니모닉(24단어)의 아날로그 봉인:** 시스템 초기 설정 시 발급되는 24개의 영어 단어는 **절대로 컴퓨터나 폰에 저장하지 마십시오.** 
    *   *방법:* 실제 종이에 볼펜으로 꾹꾹 눌러 적어 금고에 보관하십시오. 디지털 공간에서 사라진 권한을 살릴 수 있는 유일한 '신의 지팡이'입니다.
*   **행동 3. 물리적 터치 집행의 습관화:** 돈을 보낼 때, 마우스 클릭 후에 반드시 **보안 USB 키의 금색 접점을 '톡' 건드려야** 합니다. 
    *   *의미:* 이 0.1초의 신체 접촉이 있는 순간에만 돈이 흐릅니다. 해커가 원격으로 화면만 보고는 절대 돈을 빼갈 수 없게 만드는 '철통 방아쇠'입니다.

#### C. 향후 보안 실행 가능성 보증
*   현재 시스템은 **'열쇠 등록 대기'** 프레임워크가 완벽히 로딩된 상태입니다. 관리자님이 키를 준비하신 후 [등록] 절차를 밟는 즉시 1인 절대 권한이 발동됩니다.
*   본 작업은 유저들의 **마이닝 모달, 니모닉 랜덤 입력 시스템**과 물리적으로 분리되어 설계되었으므로, 유저 서비스 운영에 0.1%의 지장도 주지 않습니다.

## 3. 토큰 총 발행량(210억) 제네시스 분할 입금 교정

### [문제점]
코인 생태계가 처음 켜지는 순간, 단 1개의 '재단 통장(BitWish-Foundation)'으로 210억 개가 모두 입금되는 심각한 오류가 박혀 있습니다.

### [개선 및 구현 방안]
*   처음 코인이 태어나는 순간, 총 발행량 210억 개를 곧장 3가지 몫으로 정확하게 찢어서 별도의 시스템 통장 주소로 각각 송금해 버립니다.
    1.  **개인채굴자 전용 통장 (65%):** 136.5억 개 송금
    2.  **파트너/상점 전용 통장 (15%):** 31.5억 개 송금
    3.  **관리/재단 관리 통장 (20%):** 42억 개 송금
*   **작업 파일 위치:** `src/engine/BitWishBlockchain.ts` 에 있는 블록체인 초기화 함수(`initialize`) 시작 로직 전부 해체 후 재결합.

## 4. KYC 승인 통과자의 15일 대기 자동 지급기 (스케줄러)

### [문제점]
채굴된 가상 포인트가 실제 코인으로 변환되어 송금되기 전, 15일간의 안전 심사 대기 기간(타임락)을 무인으로 체크하고 발송하는 장치가 없습니다.

### [개선 및 구현 방안]
*   **딱지표 붙이고 대기실 이동 (DB 저장):** 관리자가 유저의 KYC 확인 승인 버튼을 누르는 즉시, 백엔드 DB 대기실에 유저 채굴량 기록과 함께 **"발송 기한: 오늘 날짜의 15일 뒤"**라는 시간 딱지를 붙여 조용히 냅둡니다.
*   **매일 자정 자동 순찰대 가동:** 메인 서버에 봇(Bot) 스크립트를 하나 심습니다. 이 로봇은 매일 밤 자정 정각마다 대기실을 순찰합니다.
*   **실제 코인 무인 발사:** 순찰 로봇이 딱지를 확인하여 **15일 기한이 도래한 유저를 발견하면**, 앞선 5번의 **'개인채굴자 통장(65%)'**에서 실제 코인을 유저의 지갑으로 자동 입금시킨 후 기록을 완료 마킹합니다.
*   **작업 파일 위치:** 백엔드 서버쪽 `cron-scheduler.ts` 파일 신규 스크립트 작성 및 몽고 DB 데이터베이스 연결.
