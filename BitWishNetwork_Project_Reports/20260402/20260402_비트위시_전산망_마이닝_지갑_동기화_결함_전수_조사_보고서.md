## 0. [전산망 분리 및 자산 침범에 관한 정밀 분석 총평]

본 분석자가 비트위시 네트워크의 서버와 클라이언트 소스코드를 전수 조사하며 내린 결론은, 현재 시스템이 **[데이터 무결성(Data Integrity)의 부재]**와 **[전산적 자산 격리(Asset Isolation)의 실패]**라는 치명적인 구조적 결함 상태에 놓여 있다는 것입니다.

### 0.1 [이해/분석] 홈페이지와 나의 지갑은 되고, 마이닝 페이지는 안 되는 이유
분석 결과, 현재 "홈페이지"와 "나의 지갑"은 `RealTimeSyncService`라는 통합 동기화 엔진을 통해 동일한 자산 필드를 공유하고 있으나, **"마이닝 페이지"는 이 전산망의 주류에서 소외된 채 독자적인 로컬 데이터(`localStorage`)에 과도하게 의존**하고 있습니다. 
서버에서 `referralRewardStorage`(1BW 보상) 데이터가 전송되더라도, 마이닝 페이지 내부의 데이터 수신 포트가 닫혀 있어 "나의 지갑"에는 숫자가 올라가는데 "마이닝 페이지"는 정지해 있는 기이한 전산 단절 현상이 발생하고 있는 것으로 분석됩니다.

### 0.2 [이해/분석] 2% 추천 보너스가 실시간으로 움직이지 않는 전산적 배경
사용자님께서 지표하신 "초당 실시간 카운팅"이 2% 보너스 보관함에서 작동하지 않는 이유는, 프론트엔드의 티커 엔진이 **[단일 수익 모델]**로만 설계되었기 때문입니다. 즉, 메인 채굴 수량만 초당 증분시키도록 하드코딩되어 있으며, 보너스 보관함은 30초마다 한 번씩 서버 응답이 올 때만 수동으로 갱신되는 보조 수단으로 전락해 있습니다. 이는 사용자님의 "가입자 즉시 반영" 철학이 소스코드 레벨에서 기술적으로 배반당하고 있음을 의미합니다.

### 0.3 [이해/분석] 관리자 초기화의 폭주와 자산 파괴의 실체
가장 치명적인 부분은 관리자 페이지의 "초기화" 버튼이 **[파괴적 덮어쓰기]** 방식으로 작동한다는 점입니다. 
전산적으로는 ‘누적 채굴량’만 리셋해야 함에도 불구하고, 현재의 로직은 **가입자 명수와 상관없이 추천 보너스율(`.referralBonusRate`)을 강제로 '0.25'로 회귀**시키고 있습니다. 이는 마치 통장의 잔고를 0으로 만들려다 계좌의 이자율과 신용 등급까지 말소해버리는 전산적 폭거이며, 이로 인해 모든 지갑의 추천인 족보가 꼬이고 자산이 증발하는 사태가 초래된 것으로 분석됩니다.

결론적으로, 현재의 시스템은 **"자산(Assets)"과 "상태(States)"를 구분하지 못하는 미숙한 설계** 단계에 머물러 있습니다. 저는 이를 바로잡아 모든 추천 정보와 보너스를 영구 자산으로 격리 보호하고, 마이닝 페이지와 지갑의 전산망을 100% 한 몸으로 묶는 공정을 제안합니다.

## 1. [핵심 결함 분석] 나의 지갑 vs 마이닝 페이지 데이터 불일치 (Issue 1.1 ~ 1.4)

### 1.1 마이닝 페이지 "추천보상보관함" 누락 및 동기화 실패 원인
- **현상:** 나의 지갑(Wallet)과 홈페이지 통계창은 실시간으로 1BW 보상을 표시하나, 마이닝 페이지는 이를 반영하지 못함.
- **근본 원인:**
    1. **UI 구현 누락:** `MiningPage.tsx` 소스코드 전수 조사 결과, `referralRewardStorage`(1BW 가입 보상)를 표시할 State와 UI 엘리먼트 자체가 설계에서 누락됨.
    2. **API 응답 필드 결여:** 실시간 동기화 API인 `/api/mining/sync`(`syncMiningData`)가 `referralBonusStorage`(2% 보너스)는 반환하지만, `referralRewardStorage`(1BW 보상) 필드를 누락하여 응답함.
- **해결 방안:** `MiningPage.tsx`에 신규 보관함 UI를 주입하고, 서버 동기화 API에 1BW 보상 필드를 강제 매핑하여 동시 업데이트 체계 구축.

### 1.2 추천 보너스(2%) 실시간 카운팅 정지 현상 (초당 카운팅 미작동)
- **현상:** 마이닝이 시작되어도 2% 추천 보너스 보관함의 숫자가 실시간(초 단위)으로 올라가지 않음.
- **근본 원인:**
    1. **티커 엔진의 단조성:** 현재 프론트엔드 티커 로직(`updateAccumulatedReward`)이 메인 채굴 수량(`accumulatedReward`)만 직접 계산하여 화면을 갱신하고, 보너스 보관함의 증분량은 30초 주기 서버 응답에만 의존하고 있음.
    2. **로컬 연산 누락:** 초당 2%의 보너스 수익률을 시각적으로 투영할 `LocalTicker` 연산이 보너스 보관함 UI 전용으로 구현되지 않음.
- **해결 방안:** 실시간 동기화 서비스(`RealTimeSyncService`)에 보너스 보너스 전용 티커 인터셉터를 주입하여 서버 응답 대기 없이 초 단위로 숫자가 올라가도록 교정.

## 2. [자산 유실 분석] 추천인 명수 및 보너스% 증발 결함 (Issue 2 ~ 2.1)

### 2.1 가입자 명수 및 보너스율 불일치 실태
- **현상:** 가입자가 발생해도 명수(`.referralCount`)가 반영되지 않거나, 반영 후에도 간헐적으로 초기화됨.
- **근본 원인:** 
    - `BonusRecord`의 `referralList` 배열 데이터와 `MiningState`의 `referralCount` 정수형 데이터가 원자적(Atomic)으로 처리되지 않고 분산되어 저장됨.
    - 특히, 가입 로직에서 추천인의 보너스율(`referralBonusRate`)을 즉시 재계산하지 않고 하드코딩된 증분값만 더하는 방식의 정밀도 하락 발생.
- **복구 대책:**
    - `BonusRecord`를 기준으로 전체 지갑의 족보를 재전수 조사하는 **[전산 족보 자동 보정 스크립트]** 가동.
    - `referralList.length`를 기반으로 `referralCount`와 `referralBonusRate`(0.02 * n)를 실시간 역추적하여 강제 보정(Override).

## 3. [관리자 결함] 무도한 초기화 로직 및 자산 파괴 방지 (Issue 3)

### 3.1 관리자 초기화(Reset)의 자산 침범 실태
- **분석 결과:** 현재 `admin.ts`와 `miningController.ts`의 리셋 로직은 "영구 자산"과 "세션 데이터"를 구분하지 못함.
- **치명적 결함 지점:**
    - `miningUpdate` 시 `$set: { currentTotalRate: "0.25" }` 강제 적용: 이로 인해 유저가 확보한 모든 추천 보너스율과 출석 보너스 권한이 즉시 몰수됨.
    - `bonusUpdate` 시 `referralBonusStorage: "0"` 강제 적용: 이미 채굴된 보너스 자산까지 0으로 덮어씀.
- **조치 계획 (자산 보호 격리벽):**
    - **격리 보호:** `accumulatedReward` (채굴 수량)와 `miningStartTime`만 리셋 대상에 포함.
    - **보존 필드 고정:** `referralCount`, `referralBonusRate`, `referralBonusStorage`, `referralRewardStorage`, `attendanceHistory`는 리셋 명령 시 `$unset`이나 `0` 처리를 절대 금지하도록 백엔드 보호 코드 주입.

## 5. [기술적 이행 상세 가이드: 파일 위치 및 핵심 로직]

비트위시 전산망의 무결성 확보를 위해 수행할 각 단계별 정밀 공정 지침이다.

### 5.1 [Back-End] DB 전수 조사 및 추천 족보 강제 복구 (Phase 1)
- **대상 파일:** `server/scripts/recalculate-all-assets.ts` (신규 전수 조사 스크립트)
- **핵심 로직:**
    1. 전 지갑의 `BonusRecord`를 순회하며 `referralList` 배열의 실제 길이를 추출한다.
    2. 추출된 길이를 기반으로 `MiningState`의 `referralCount`(가입자 수)와 `referralBonusRate`(길이 * 0.02)를 재계산한다.
    3. `isAttendanceActive`가 `true`일 경우에만 5% 보너스를 더해 `currentTotalRate`를 최종 업데이트하여, DB 상의 누락된 숫자를 실제 족보 데이터와 일치시킨다.

### 5.2 [Back-End] 관리자 초기화 로직 의 자산 보호벽 주입 (Phase 2)
- **대상 파일 1:** `server/routes/admin.ts` (POST /api/admin/mining/reset)
    - **수정 로직:** `miningUpdate`의 `$set` 구문에서 `referralCount`, `referralBonusRate`, `currentTotalRate`를 제거한다. 이를 통해 초기화 시 사용자의 마이닝 권한(보너스율)이 '0.25'로 회귀하는 현상을 원천 차단한다.
- **대상 파일 2:** `server/controllers/miningController.ts` (`resetMiningData` 함수)
    - **수정 로직:** 초기화 매개변수 중 마이닝률 관련 필드를 삭제하고, 오직 `accumulatedReward`와 `miningStartTime`만 초기화하도록 격리한다.

### 5.3 [Front-End] 전산망 통합 및 실시간 보증 티커 구현 (Phase 3)
- **대상 파일 1:** `server/controllers/miningController.ts` (`syncMiningData` 및 `getUserStatus`)
    - **수정 로직:** 응답 JSON 데이터에 `referralRewardStorage`(1BW 보상) 필드를 추가 주입하여 지갑과 마이닝 페이지가 동일한 필드명을 수신하도록 통일한다.
- **대상 파일 2:** `src/components/MiningPage/MiningPage.tsx`
    - **수정 로직:** 
        1. `referralRewardStorage` 상태 변수를 추가하고 서버로부터 수신된 보관함 데이터를 바인딩한다.
        2. `updateAccumulatedReward` 함수 내부에 `referralBonusRate`를 기반으로 한 **보너스 전용 초당 증분 로직**을 추가하여, 2% 보너스 숫자가 실시간으로 올라가는 시각적 무결성을 확보한다.
