## 1. [기술 분석] 왜 기존 시스템은 "사기"였으며, 어디를 고쳐야 하는가?

### (1) 기존 시스템의 결함 분석 (Root Causes)
*   **분석 지점:** `server/controllers/UserController.ts`
*   **문제:** 추천 발생 시 단순히 수치(`referralCount`)만 올리고, 정작 사용자님이 원하시는 **[내역 기록]** 로직이 아예 존재하지 않습니다.
*   **파일별 수정 계획:**
    1.  **`server/models/BonusRecord.ts`**: 내역 정보를 담을 `HistoryLog` 서브 스키마 추가 (지갑 주소, 시간, 6가지 상태값 저장).
    2.  **`server/controllers/UserController.ts`**: 가입 즉시 로그를 생성하여 개인 DB 저장소에 삽입하는 `pushlog` 로직 이식.
    3.  **`src/components/MyWalletModal/MyWalletModal.tsx`**: "Coming Soon" 더미 텍스트를 제거하고, `activeTab === 'referralRewards'` 시 개별 유저의 `referralList` 데이터를 호출하는 전용 API 연동.

## 2. [작업 계획 플랜] 단계별 정밀 집행 가이드

### **[Step 1] DB 스키마 확장 및 개별 저장소 확립 (Back-end)**
*   **작업 내용:** 모든 유저는 본인의 `BonusRecord` 내에 독립된 `referralList` 배열을 가집니다. 여기에 단순히 주소만 담는 게 아니라 **[보상 지급 여부(Status), 보너스 부여 여부(RateStatus), 실시간 KYC 현황]** 필드를 추가합니다.
*   **기대효과:** 전역 변수 없이 유저 개개인의 데이터 저장소에서만 데이터를 읽어오는 보안성과 독립성 확보.

### **[Step 2] 엔진 정상화 및 소급 적용 (Data Normalization)**
*   **작업 내용:** 현재 0%로 방치된 모든 가입자를 전수 조사하는 `normalization_script.js`를 실행합니다.
*   **로직:** `User`의 `referrerCode`가 존재함에도 `MiningState.referralBonusRate`가 0인 유저를 추출 -> 일괄 **2%(0.02) 강제 주입** 및 **1BW 보상 유무 확인** 로직 수행.
*   **기대효과:** 유저 2(`...7A626`)를 포함한 모든 낙오 유저의 권리 즉시 복원.

### **[Step 3] 가입 즉시 내역 등록 자동화 (Logging Engine)**
*   **작업 내용:** 추천인이 가입 버튼을 누르는 즉시, 보유자의 DB에 다음 정보를 자동 생성하여 삽입합니다.
    *   `childAddress`: 가입자 주소
    *   `timestamp`: `YYYY.MM.DD HH:mm:ss`
    *   `rewardStatus`: `PAID` (Blue) or `PENDING` (Red)
    *   `bonusStatus`: `ACTIVE` (Blue) or `INACTIVE` (Red)
*   **기대효과:** 사용자님이 말씀하신 "가입 즉시 등록" 원칙의 완전한 시스템 구현.

### **[Step 4] P2P 지갑 내역 UI 정밀 이식 (Front-end)**
*   **작업 내용:** `MyWalletModal.tsx` 내 추천 보너스 탭을 구현합니다.
*   **구현 상세:**
    *   상단 파란색/빨간색 색상 가이드 텍스트 배치.
    *   테이블 컬럼: 주소 | 시간 | 1BW 보상 | 2% 보너스 | KYC 소식.
    *   **P2P 침범 금지:** 기존 P2P 송금/조회 로직 소스 코드는 단 한 줄도 수정하지 않고, 오직 새로운 탭 컴포넌트만 추가하여 독립성 유지.

## 3. [최종 기대효과] 진실된 시스템의 모습

1.  **투명성:** 유저가 본인의 지갑에서 누가 언제 가입해서 보너스를 줬는지 파란색/빨간색으로 즉시 확인 가능.
2.  **미래 보장:** 앞으로 백만 명의 가입자가 발생해도, 시스템은 가입 즉시 위 로직을 반복하며 오차 없이 내역을 누적함.
3.  **신뢰성:** 수동으로 수치를 고치는 것이 아닌, 실제 가입 내역(Log)이 존재하는 상태에서 수치가 산출되는 역추적 가능한 시스템 구축.
