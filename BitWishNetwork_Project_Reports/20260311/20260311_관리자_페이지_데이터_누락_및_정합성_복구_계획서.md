# [2026-03-11] 관리자 페이지 데이터 누락 및 정합성 복구 계획서

**작성일자:** 2026년 3월 11일
**주요 목적:** 관리자 페이지 가입자 목록 누락(7/12명) 해결 및 채굴 데이터 일치

---

## 1. 장애 원인 분석 (Admin Page 집중)

### **(1) 가입자 목록 12개 중 7개만 표기되는 원인**
- **팩트:** UI 상단에서 날짜 선택 드롭다운은 제거되었으나, 내부 소스 코드에서는 여전히 현재 날짜(2026년 3월) 파라미터를 서버로 전송 중.
- **백엔드 현황:** 서버의 가입자 조회 API는 날짜 값이 넘어올 경우 '해당 월에 가입한 유저는 7명'으로 인식하여 필터링된 결과만 반환함.

### **(2) 전체 채굴량 합산 불일치**
- **팩트:** 현재 목록에 보이는 7명에 대한 채굴 데이터만 합산되고 있어, 과거 가입자 5명의 채굴량이 통계에서 누락됨.

---

## 2. 세부 복구 프로세스

### **Step 1. 백엔드 필터링 전면 해제 (`admin.ts`)**
- **파일 경로:** `server/routes/admin.ts`
- **조치 사항:** 
    - `/api/admin/referral/all` 및 관련 API에서 날짜 필터(`year`, `month`)를 강제로 무시하거나, 파라미터가 없을 때 무조건 전체 유저를 반환하도록 쿼리 수정.
    - DB 내 `MiningState` 컬렉션의 모든 데이터를 가입 일자와 상관없이 12명 전체를 긁어오도록 보장.

### **Step 2. 관리자 페이지 호출 파라미터 제거 (`AdminPage.tsx`)**
- **파일 경로:** `src/components/AdminPage/AdminPage.tsx`
- **조치 사항:**
    - `fetchReferralData` 호출 시 날짜 쿼리 스트링(`?year=...&month=...`)을 삭제.
    - 탭 진입 즉시 12명 전체 데이터가 화면에 출력되도록 호출 루틴 최적화.

## 4. [추가 분석] 가입자 실종(3명) 및 중복 데이터 원인 보고

사용자님께서 제보하신 **"9명만 반영되고 3명이 누락된 현상"** 및 **"특정 주소 중복 노출"** 건에 대한 정밀 분석 결과입니다.

### **(1) 가입자 3명 실종 원인**
- **대상:** `BW9F5F...`, `BWD6CC...`, `BW29DF...` (총 3명)
- **원인:** 해당 유저들은 회원가입(`User`)은 완료되었으나, **한 번도 채굴 시작 버튼을 누르지 않아** `MiningState` 문서가 생성되지 않은 상태입니다.
- **결함:** 제가 Step 1에서 수정한 코드가 `MiningState`를 기점으로 긁어오도록 짜여 있어, 채굴 기록이 없는 유저는 조회 대상에서 아예 누락되었습니다.

### **(2) 특정 지갑 중복 노출 원인**
- **대상:** `BW070DEAB46086F60451C35A5C2CF54C4F0F255E65`
- **원인:** 과거 시스템 불안정 시기에 동일 지갑 주소에 대한 `MiningState` 문서가 2개 중복 생성되었습니다. 이로 인해 목록에 두 줄로 출력되는 것입니다.

### **(3) 수리 계획 (Repair Strategy)**
- **조회 원천 변경:** `MiningState`가 아닌 **`User` 컬렉션을 원천(Base)**으로 12명 전체를 먼저 불러온 뒤, `MiningState` 데이터를 붙이는(Left Join) 방식으로 코드를 재수정하겠습니다.
- **결과:** 이렇게 하면 채굴을 시작하지 않은 3명도 '0.00 BW'로 목록에 정확히 표시되며, 전체 12명 라인업이 완벽히 복구됩니다.
- **데이터 정리:** 중복된 `BW070DEAB...` 데이터는 1개만 남기고 정제 처리하겠습니다.
