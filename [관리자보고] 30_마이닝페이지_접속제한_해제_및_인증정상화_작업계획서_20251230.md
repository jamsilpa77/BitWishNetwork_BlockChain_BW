## 1. 작업 목적 및 배경

현재 시스템은 보안을 위해 "1기기 1지갑" 원칙을 고수하여, 기기에 등록된 지갑이 아닐 경우 마이닝 인증 모달에서 접속을 차단하고 있습니다. 하지만 이는 관리자님의 테스트 편의성을 저하시키고, 2차 비밀번호를 정상적으로 설정한 유저의 접근성을 제한하는 요소가 됩니다.

본 작업은 **이미 구현 완료된 [개인 단독 저장소(BISE)] 엔진의 무결성을 완벽히 유지**하면서, 어떤 지갑 주소라도 2차 비밀번호만 일치하면 마이닝 페이지를 즉시 열 수 있도록 인증 로직을 정상화하는 데 목적이 있습니다.

---

## 2. 지시 사항 정밀 분석 및 기술적 대응

### 가. 기기 점유 지갑 제약 해제 (Frontend)
- **명령:** "모든 지갑 주소가 2차 비밀번호 등록이 끝났다면 마이닝 페이지를 열 수 있도록 수정"
- **대응:** `MiningAuthModal.tsx`에서 로컬 스토리지의 지갑 주소와 입력된 주소를 비교하여 차단하는 **"접속 지갑 불일치" 안전 장치를 삭제**합니다.
- **효과:** 주체 지갑과 상관없이 인증 절차를 거친 모든 유효 지갑의 마이닝 상태를 즉시 확인할 수 있습니다.

### 나. 개인 단독 저장소 기반 인증 (Backend 연동)
- **보안:** 비밀번호 해시는 이전 작업에서 구현된 **PBKDF2 SHA512 (100,000회)** 방식을 그대로 유지합니다.
- **매칭:** 입력된 지갑 주소에 해당하는 **개별 데이터 파일(`.json`)**을 서버에서 로드하여, 저장된 해시값과 대조합니다. 
- **안전성:** 이 과정에서 어떠한 데이터 쓰기도 발생하지 않으므로, 기존의 보상/보너스 데이터는 100% 안전하게 보호됩니다.

### 다. 다계정 방지 전략 (Line 372 지시사항 반영)
- **현재:** 테스트 편의를 위해 로컬 차단 로직은 제거합니다.
- **차후:** 모든 통합 기능이 완성된 후, 지시하신 대로 **"확장 노드 팝업"** 형태의 다계정 방지 시스템을 별도 설치하여 정식 보안을 확보합니다.

## 3. 상세 수정 단계 (Phase 1 ~ 2)

### Phase 1: `MiningAuthModal.tsx` 로직 개편
1.  **제약 조건 제거:** `currentStoredAddress`를 가져와서 입력 주소와 대조하는 if문(라인 70~81 부근)을 통째로 도려냅니다.
2.  **인증 요청 일원화:** 주소가 무엇이든 곧바로 서버의 `api/user/status` 또는 인증 엔드포인트에 비밀번호 검증을 요청하도록 수정합니다.
3.  **성공 후 처리:** 인증 성공 시 `onSuccess(address)`를 호출하여 마이닝 상태 모달에 해당 주소를 올바르게 전달합니다.

### Phase 2: 서버-클라이언트 인증 동기화 및 즉시 접속 체계 완성
1.  **중복 설정창 박멸:** 서버(`server.js`)의 `/api/user/status` 응답과 프론트엔드(`WalletService.ts`)의 `hasSecondPassword` 로직을 분자 단위로 일치시켜, 이미 비밀번호가 있는 유저에게 자꾸 설정 창이 뜨는 현상을 원천 제거합니다.
2.  **즉시 접속 로직:** 지갑 주소와 비밀번호 입력 후 [확인] 클릭 시, 어떠한 추가 확인 메시지나 설정창 유도 없이 **즉시 마이닝 페이지로 이동**하도록 프로세스를 간소화합니다.
3.  **데이터 무결성 검증:** 읽기(Read) 전용 인증 프로세스를 강화하여 파일 시스템 마찰을 0%로 유지하고, 비밀번호 해시가 훼손되지 않도록 보존합니다.
4.  **5인 실검증:** 관리자, 가입자1~4 주소를 번갈아 입력하며 일체의 지체나 오류 없이 각자의 보관함 수치(reward/bonus Storage)가 정상 출력되는지 최종 확인합니다.
